# Chapter08

## 1. 인증(Authentication)과 인가(Authorization)

웹 서비스에서는 사용자가 누구인지 확인하고, 그 사용자가 어떤 기능을 사용할 수 있는지 판단해야 한다.

이때 사용하는 개념이 **인증(Authentication)**과 **인가(Authorization)**이다.

두 개념은 비슷해 보이지만 역할이 다르다.

```text
인증(Authentication)
→ 사용자가 누구인지 확인하는 과정

인가(Authorization)
→ 인증된 사용자가 어떤 기능을 사용할 권한이 있는지 확인하는 과정
```

### 1.1 인증이란?

인증(Authentication)이란 **서비스를 이용하려는 사용자가 누구인지 확인하는 과정**이다.

쉽게 표현하면 다음과 같다.

```text
인증 = “너 누구야?”
```

가장 대표적인 인증 방법은 로그인이다.

사용자가 이메일과 비밀번호를 입력하면 서버는 데이터베이스에 저장된 회원 정보와 비교한다.

```text
사용자가 이메일과 비밀번호 입력
→ 서버가 이메일로 회원 조회
→ 입력한 비밀번호와 저장된 비밀번호 비교
→ 정보가 일치하면 인증 성공
→ 정보가 일치하지 않으면 인증 실패
```

예를 들어 사용자가 다음과 같은 정보로 로그인한다고 가정한다.

```json
{
  "email": "user@email.com",
  "password": "12345678"
}
```

서버는 먼저 `user@email.com`이라는 이메일을 가진 회원이 존재하는지 확인한다.

회원이 존재한다면 사용자가 입력한 비밀번호와 데이터베이스에 저장된 비밀번호가 일치하는지 확인한다.

두 정보가 모두 일치하면 서버는 다음과 같이 판단할 수 있다.

```text
이 요청을 보낸 사용자는 실제로 가입된 회원이다.
```

이 과정이 인증이다.

인증의 대표적인 예시는 다음과 같다.

- 이메일과 비밀번호를 이용한 로그인
- 휴대전화 인증
- 이메일 인증
- 소셜 로그인
- 지문 또는 얼굴을 이용한 생체 인증
- 세션을 이용한 사용자 확인
- 토큰을 이용한 사용자 확인

### 1.2 인가란?

인가(Authorization)란 **인증된 사용자가 특정 페이지, API, 기능 또는 데이터에 접근할 권한이 있는지 확인하는 과정**이다.

쉽게 표현하면 다음과 같다.

```text
인가 = “너 이거 해도 돼?”
```

사용자가 로그인에 성공했다고 해서 서비스의 모든 기능을 사용할 수 있는 것은 아니다.

예를 들어 일반 사용자와 관리자가 있다고 가정한다.

```text
일반 사용자
- 자신의 정보 조회
- 게시글 작성
- 자신의 게시글 수정 및 삭제

관리자
- 모든 회원 정보 조회
- 모든 게시글 관리
- 관리자 페이지 접근
```

일반 사용자가 관리자 페이지에 접근하려고 하면 서버는 다음을 확인한다.

```text
이 사용자가 로그인했는가?
→ 인증 확인

이 사용자가 관리자 권한을 가지고 있는가?
→ 인가 확인
```

일반 사용자는 로그인에는 성공했기 때문에 인증된 사용자이다.

하지만 관리자 권한은 없으므로 관리자 페이지에 접근할 수 없다.

```text
일반 사용자 로그인
→ 인증 성공
→ 관리자 페이지 요청
→ 관리자 권한 확인
→ 관리자 권한 없음
→ 접근 거부
```

인가는 페이지에만 적용되는 것이 아니다.

다음과 같은 경우에도 인가가 필요하다.

- 관리자 페이지 접근
- 관리자 전용 API 호출
- 다른 사용자의 게시글 삭제
- 다른 사용자의 개인정보 조회
- 특정 등급의 회원만 사용할 수 있는 기능
- 유료 회원 전용 콘텐츠 접근

즉, 인가는 다음과 같은 범위에 적용된다.

```text
페이지 접근 권한
API 호출 권한
기능 사용 권한
데이터 조회 및 수정 권한
```

### 1.3 인증과 인가의 차이

인증과 인가는 모두 보안과 관련된 개념이지만 확인하는 대상이 다르다.

| 구분    | 인증                | 인가                     |
| ----- | ----------------- | ---------------------- |
| 영어    | Authentication    | Authorization          |
| 의미    | 사용자가 누구인지 확인      | 사용자가 어떤 권한을 가지고 있는지 확인 |
| 핵심 질문 | 너 누구야?            | 너 이 기능 사용해도 돼?         |
| 대표 예시 | 로그인               | 관리자 페이지 접근             |
| 확인 정보 | 이메일, 비밀번호, 세션, 토큰 | 역할과 권한                 |
| 처리 순서 | 인가보다 먼저 처리        | 인증 후 처리                |
| 실패 예시 | 로그인 실패, 토큰 만료     | 관리자 권한 없음              |

여기서 인증이 먼저 처리되고 인가가 인증 후 처리된다는 것은 **인증이 인가보다 먼저 선행된다는 뜻**이다.

서버가 사용자의 권한을 확인하려면 먼저 그 사용자가 누구인지 알아야 한다.

```text
1. 사용자가 누구인지 확인
→ 인증

2. 확인된 사용자가 어떤 권한을 가지고 있는지 확인
→ 인가
```

학교를 예로 들면 다음과 같다.

```text
학생증을 확인하여 학생이 누구인지 확인
→ 인증

학생이 특정 연구실에 들어갈 권한이 있는지 확인
→ 인가
```

학생이 누구인지도 모르는 상태에서는 그 학생이 어떤 권한을 가지고 있는지 판단하기 어렵다.

따라서 일반적으로 인증이 먼저 이루어지고, 인증된 사용자를 대상으로 인가가 진행된다.

한 문장으로 정리하면 다음과 같다.

> 먼저 사용자가 누구인지 확인하고, 그다음 그 사용자가 무엇을 할 수 있는지 확인한다.

### 1.4 인증과 인가의 처리 순서

일반적인 웹 서비스의 인증과 인가 과정은 다음과 같다.

```text
1. 사용자가 로그인 요청을 보낸다.

2. 서버가 사용자가 입력한 정보를 확인한다.

3. 이메일과 비밀번호가 일치하면 인증에 성공한다.

4. 서버가 세션 또는 토큰을 발급한다.

5. 사용자는 이후 API를 요청할 때 세션 정보 또는 토큰을 함께 보낸다.

6. 서버가 세션 또는 토큰을 확인하여 사용자가 누구인지 판단한다.

7. 서버가 해당 사용자의 권한을 확인한다.

8. 필요한 권한이 있으면 요청을 처리한다.

9. 필요한 권한이 없으면 요청을 거부한다.
```

전체 흐름을 간단히 표현하면 다음과 같다.

```text
로그인 요청
→ 사용자 정보 확인
→ 인증 성공
→ 세션 또는 토큰 발급
→ API 요청
→ 인증 상태 확인
→ 권한 확인
→ Controller 실행 또는 접근 거부
```

예를 들어 일반 사용자가 자신의 정보를 조회하는 경우는 다음과 같다.

```text
사용자 로그인
→ 인증 성공
→ 토큰 발급
→ /user/me API 요청
→ 토큰 확인
→ 인증된 사용자임을 확인
→ 로그인한 사용자만 접근 가능한 API인지 확인
→ 접근 허용
→ 내 정보 반환
```

관리자가 회원 목록을 조회하는 경우는 다음과 같다.

```text
관리자 로그인
→ 인증 성공
→ 토큰 발급
→ /admin/members API 요청
→ 토큰 확인
→ 관리자 사용자임을 확인
→ 관리자 권한 확인
→ 접근 허용
→ 회원 목록 반환
```

일반 사용자가 관리자 API를 요청하는 경우는 다음과 같다.

```text
일반 사용자 로그인
→ 인증 성공
→ /admin/members API 요청
→ 사용자가 누구인지 확인
→ 일반 사용자 권한임을 확인
→ 관리자 권한 없음
→ 접근 거부
```

이 경우 사용자가 로그인했기 때문에 인증에는 성공했지만, 관리자 권한이 없기 때문에 인가에는 실패한 것이다.

### 1.5 세션이란?

세션(Session)은 **서버가 사용자의 로그인 상태를 저장하고 기억하는 방식**이다.

사용자가 로그인에 성공하면 서버는 해당 사용자의 정보를 서버 내부에 저장한다.

그리고 저장된 정보를 찾을 수 있도록 세션 ID를 발급한다.

```text
사용자 로그인 성공
→ 서버가 세션 생성
→ 서버가 로그인 정보 저장
→ 세션 ID 발급
→ 브라우저가 세션 ID 저장
```

서버에는 다음과 같은 정보가 저장될 수 있다.

```text
세션 ID: ABC123
회원 ID: 1
이름: 김경민
권한: ROLE_USER
```

사용자는 이후 요청마다 세션 ID를 서버에 보낸다.

```text
사용자 API 요청
→ 세션 ID ABC123 전달
→ 서버가 ABC123 세션 조회
→ 회원 ID 1번 사용자임을 확인
→ 요청 처리
```

세션 ID는 일반적으로 브라우저의 쿠키에 저장된다.

브라우저는 같은 서버에 요청을 보낼 때 쿠키를 자동으로 함께 보낸다.

```text
브라우저
→ 쿠키에 세션 ID 저장
→ 요청마다 세션 ID 자동 전송
```

세션을 쉽게 비유하면 식당의 번호표와 비슷하다.

```text
서버
→ 번호표 10번은 김경민 사용자라고 기억함

사용자
→ 요청할 때마다 번호표 10번을 보여줌
```

세션의 핵심은 다음과 같다.

> 로그인 정보는 서버가 저장하고, 사용자는 세션 ID를 가지고 다닌다.

### 1.6 토큰이란?

토큰(Token)은 **로그인한 사용자임을 증명하기 위해 서버가 발급하는 인증용 문자열**이다.

사용자가 로그인에 성공하면 서버는 토큰을 만들어 사용자에게 전달한다.

```text
사용자 로그인 성공
→ 서버가 토큰 발급
→ 사용자가 토큰 저장
→ API 요청마다 토큰 전송
```

토큰은 다음과 같이 긴 문자열 형태로 만들어질 수 있다.

```text
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxIn0...
```

사용자는 API를 요청할 때 HTTP 요청 헤더에 토큰을 포함한다.

```http
GET /user/me
Authorization: Bearer access-token
```

서버는 전달받은 토큰을 검증한다.

```text
토큰이 정상임
→ 인증된 사용자로 판단
→ 요청 처리

토큰이 없음
→ 인증되지 않은 사용자로 판단
→ 요청 거부

토큰이 만료됨
→ 인증 실패
→ 요청 거부

토큰이 위조되거나 잘못됨
→ 인증 실패
→ 요청 거부
```

토큰을 쉽게 비유하면 출입증과 비슷하다.

```text
서버가 사용자에게 출입증을 발급
→ 사용자가 출입증을 보관
→ 기능을 사용할 때마다 출입증을 보여줌
```

토큰의 핵심은 다음과 같다.

> 사용자가 서버에서 발급받은 인증용 문자열을 보관하고, 요청할 때마다 서버에 전달한다.

### 1.7 세션과 토큰의 차이

세션과 토큰은 모두 로그인한 사용자를 계속 확인하기 위한 방법이다.

로그인 이후 사용자가 API를 요청할 때마다 이메일과 비밀번호를 다시 보내는 것은 불편하고 위험하다.

따라서 로그인에 성공하면 세션 ID나 토큰을 발급하고, 이후 요청에서는 그것을 이용하여 사용자를 확인한다.

| 구분          | 세션       | 토큰               |
| ----------- | -------- | ---------------- |
| 로그인 정보 저장   | 서버에 저장   | 일반적으로 토큰을 이용해 확인 |
| 사용자가 보관하는 것 | 세션 ID    | 토큰               |
| 요청 시 전달     | 세션 ID    | 토큰               |
| 대표적인 저장 위치  | 브라우저 쿠키  | 쿠키 또는 클라이언트 저장소  |
| 서버의 상태 저장   | 필요       | 일반적으로 필요하지 않음    |
| 대표적인 방식     | 세션 기반 인증 | JWT 기반 인증        |

쉽게 비교하면 다음과 같다.

```text
세션
→ 서버가 사용자를 기억한다.

토큰
→ 사용자가 인증 정보를 나타내는 출입증을 가지고 다닌다.
```

### 1.8 상황별 세션과 토큰 사용

세션과 토큰 중 하나가 무조건 더 좋은 것은 아니다.

서비스의 구조와 목적에 따라 적절한 방식을 선택해야 한다.

#### 세션을 주로 사용하는 상황

세션은 전통적인 웹사이트나 서버와 화면이 하나의 애플리케이션으로 구성된 경우에 많이 사용한다.

예시는 다음과 같다.

- Spring Boot가 HTML 화면까지 함께 제공하는 서비스
- 회사 내부 관리 시스템
- 학교 홈페이지
- 관리자 페이지
- 서버 규모가 크지 않은 웹 서비스
- 로그아웃을 즉시 처리하는 것이 중요한 서비스

세션 방식에서는 서버가 로그인 상태를 가지고 있기 때문에 사용자가 로그아웃하면 서버에 저장된 세션을 삭제하면 된다.

```text
사용자 로그아웃
→ 서버의 세션 삭제
→ 기존 세션 ID 사용 불가
```

세션이 잘 맞는 구조는 다음과 같다.

```text
브라우저
↔ 하나의 Spring Boot 서버
```

#### 토큰을 주로 사용하는 상황

토큰은 프론트엔드와 백엔드가 분리된 서비스에서 많이 사용한다.

예시는 다음과 같다.

- React + Spring Boot
- Next.js + Spring Boot
- 모바일 앱 + Spring Boot
- REST API 중심 서비스
- 여러 개의 서버를 사용하는 서비스
- 프론트엔드와 백엔드를 각각 배포하는 서비스

토큰이 잘 맞는 구조는 다음과 같다.

```text
React 또는 Next.js 프론트엔드
→ Spring Boot API 서버
→ 데이터베이스
```

예를 들어 프론트엔드를 Vercel에 배포하고, Spring Boot 백엔드를 Render나 Railway에 배포하는 구조라면 JWT 같은 토큰 방식을 많이 사용한다.

상황별로 간단히 정리하면 다음과 같다.

| 상황                         | 주로 사용하는 방식 |
| -------------------------- | ---------- |
| Spring Boot가 화면과 서버를 모두 담당 | 세션         |
| React와 Spring Boot가 분리됨    | 토큰         |
| 모바일 앱 로그인                  | 토큰         |
| 회사 내부 관리자 사이트              | 세션         |
| REST API 중심 서비스            | 토큰         |
| 여러 서버로 확장할 예정              | 토큰         |
| 로그아웃 시 즉시 인증을 종료하고 싶음      | 세션이 비교적 편리 |

### 1.9 로그인 예시

사용자가 이메일과 비밀번호를 이용하여 로그인하는 과정을 살펴보면 다음과 같다.

#### 1단계: 로그인 요청

```http
POST /auth/login
Content-Type: application/json
```

```json
{
  "email": "user@email.com",
  "password": "12345678"
}
```

#### 2단계: 회원 조회

서버는 이메일을 이용하여 회원을 조회한다.

```java
Member member = memberRepository.findByEmail(request.getEmail())
        .orElseThrow(() -> new MemberException(MEMBER_NOT_FOUND));
```

회원이 존재하지 않으면 인증에 실패한다.

#### 3단계: 비밀번호 확인

실제 서비스에서는 비밀번호를 그대로 저장하지 않고 암호화하여 저장한다.

사용자가 입력한 비밀번호와 저장된 암호화 비밀번호를 비교한다.

```java
if (!passwordEncoder.matches(
        request.getPassword(),
        member.getPassword()
)) {
    throw new MemberException(INVALID_PASSWORD);
}
```

비밀번호가 일치하지 않으면 인증에 실패한다.

#### 4단계: 인증 성공

이메일과 비밀번호가 모두 일치하면 인증에 성공한다.

서버는 세션을 만들거나 토큰을 발급한다.

```json
{
  "isSuccess": true,
  "code": "COMMON200_1",
  "message": "로그인에 성공했습니다.",
  "result": {
    "memberId": 1,
    "accessToken": "example-access-token"
  }
}
```

#### 5단계: 인증이 필요한 API 요청

사용자는 로그인할 때 받은 토큰을 요청 헤더에 포함한다.

```http
GET /user/me
Authorization: Bearer example-access-token
```

서버는 토큰을 검증하여 사용자를 확인한다.

```text
토큰이 유효함
→ 인증된 사용자
→ 요청 처리

토큰이 없거나 유효하지 않음
→ 인증 실패
→ 요청 거부
```

관리자 API라면 인증 후에 관리자 권한까지 확인한다.

```text
토큰으로 사용자 확인
→ 인증 성공
→ 사용자 권한 확인
→ ROLE_ADMIN이면 접근 허용
→ ROLE_USER이면 접근 거부
```

## 2. Spring Security

### 2.1 Spring Security란?

Spring Security는 **Spring 애플리케이션의 인증, 인가 및 여러 보안 기능을 처리하는 프레임워크**이다.

쉽게 말하면 사용자의 요청이 Controller에 도착하기 전에 다음 내용을 검사하고 처리한다.

```text
이 요청을 보낸 사용자가 누구인가?
→ 인증

이 사용자가 해당 기능을 사용할 권한이 있는가?
→ 인가
```

Spring Security는 단순히 인증과 인가 결과를 확인하기만 하는 것이 아니다.

다음과 같은 역할을 실제로 수행한다.

- 로그인 요청에서 사용자 정보를 확인한다.
- 비밀번호가 올바른지 검증한다.
- 세션이나 토큰이 유효한지 검증한다.
- 현재 요청의 사용자를 인증된 사용자로 등록한다.
- 사용자가 가진 권한을 확인한다.
- URL 또는 API 접근을 허용하거나 거부한다.
- 인증에 실패하면 요청을 차단한다.
- 권한이 부족하면 요청을 차단한다.

즉, Spring Security는 인증과 인가를 **검사하면서 실제 접근 허용 여부까지 결정하는 역할**을 한다.

### 2.2 Spring Security가 필요한 이유

Spring Security를 사용하지 않는다면 각 Controller에서 직접 로그인 여부와 권한을 확인해야 한다.

```java
@GetMapping("/admin/members")
public String getMembers() {

    // 사용자가 로그인했는지 확인
    // 사용자의 세션 또는 토큰 확인
    // 관리자 권한인지 확인
    // 권한이 없으면 예외 발생

    return "회원 목록";
}
```

API가 많아질수록 같은 코드가 반복된다.

```text
/user/me
→ 로그인 확인 코드 필요

/posts
→ 로그인 확인 코드 필요

/admin/members
→ 로그인 확인 + 관리자 권한 확인 코드 필요
```

이런 방식은 코드의 중복이 많아지고 보안 설정을 놓칠 가능성도 커진다.

Spring Security를 사용하면 접근 규칙을 한곳에서 관리할 수 있다.

```java
.requestMatchers("/auth/**").permitAll()
.requestMatchers("/admin/**").hasRole("ADMIN")
.anyRequest().authenticated()
```

위 설정은 다음과 같은 의미이다.

```text
/auth/**
→ 누구나 접근 가능

/admin/**
→ 관리자만 접근 가능

그 외 요청
→ 로그인한 사용자만 접근 가능
```

Spring Security를 사용하면 다음과 같은 장점이 있다.

- 인증과 인가를 한곳에서 관리할 수 있다.
- Controller의 반복 코드를 줄일 수 있다.
- 로그인하지 않은 사용자의 접근을 차단할 수 있다.
- 사용자 권한에 따라 API 접근을 제한할 수 있다.
- 비밀번호를 안전하게 암호화할 수 있다.
- 인증 실패와 권한 실패를 일관되게 처리할 수 있다.

### 2.3 Spring Security의 요청 처리 흐름

Spring Security는 사용자의 요청이 Controller에 도착하기 전에 먼저 요청을 검사한다.

```text
사용자 요청
→ Spring Security Filter
→ 인증 처리
→ 인가 처리
→ Controller
→ 응답
```

자세한 처리 흐름은 다음과 같다.

```text
1. 사용자가 API 요청을 보낸다.

2. Spring Security Filter가 요청을 먼저 받는다.

3. 요청에 세션 ID나 토큰이 있는지 확인한다.

4. 세션 또는 토큰이 유효한지 검증한다.

5. 유효하다면 사용자가 누구인지 확인하고 인증된 사용자로 등록한다.
   → 인증 처리

6. 해당 사용자가 요청한 API에 접근할 권한이 있는지 검사한다.
   → 인가 처리

7. 필요한 권한이 있으면 Controller를 실행한다.

8. 인증 정보가 잘못되었거나 권한이 없으면 요청을 거부한다.
```

처음에는 Spring Security가 확인만 하는 것처럼 보일 수 있다.

하지만 여기서 말하는 확인은 단순한 조회가 아니다.

```text
세션 또는 토큰 확인
→ 정보가 유효한지 검증
→ 사용자가 누구인지 판단
→ 인증 정보 생성 및 저장
```

Spring Security는 확인 결과를 바탕으로 현재 요청을 인증된 요청으로 처리할지 결정한다.

인가 과정에서도 단순히 권한만 확인하는 것이 아니다.

```text
사용자 권한 확인
→ API에 필요한 권한과 비교
→ 접근 허용 또는 접근 거부 결정
```

따라서 Spring Security는 인증과 인가를 실제로 처리하는 역할을 한다.

### 2.4 로그인할 때의 인증 처리

사용자가 처음 로그인할 때는 이메일과 비밀번호를 이용하여 인증을 진행한다.

```text
로그인 요청
→ 이메일로 회원 조회
→ 비밀번호 비교
→ 일치하면 인증 성공
→ 일치하지 않으면 인증 실패
```

인증에 성공하면 Spring Security는 사용자의 인증 정보를 관리할 수 있다.

인증 정보에는 다음과 같은 내용이 포함될 수 있다.

```text
사용자 ID
이메일
권한
인증 성공 여부
```

예를 들면 다음과 같다.

```text
사용자: user@email.com
권한: ROLE_USER
인증 여부: true
```

Spring Security에서는 이러한 인증 정보를 주로 `Authentication` 객체로 표현한다.

### 2.5 로그인 이후의 인증 처리

로그인 이후 사용자가 API를 요청할 때마다 이메일과 비밀번호를 다시 보내지는 않는다.

대신 세션 ID나 토큰을 전달한다.

```http
GET /user/me
Authorization: Bearer access-token
```

Spring Security는 요청에 포함된 인증 정보를 확인한다.

```text
토큰 추출
→ 토큰 유효성 검증
→ 토큰에서 사용자 정보 확인
→ 인증된 사용자로 등록
```

세션 방식이라면 다음과 같이 동작한다.

```text
세션 ID 확인
→ 서버에 저장된 세션 조회
→ 로그인한 사용자 정보 확인
→ 인증된 사용자로 처리
```

토큰이나 세션이 유효하지 않으면 인증에 실패한다.

```text
토큰 없음
토큰 만료
잘못된 토큰
존재하지 않는 세션
만료된 세션
→ 인증 실패
```

### 2.6 Spring Security의 인가 처리

인증이 완료되면 Spring Security는 사용자가 해당 API에 접근할 권한이 있는지 확인한다.

예를 들어 다음과 같은 접근 규칙이 있다고 가정한다.

```java
.requestMatchers("/admin/**").hasRole("ADMIN")
.requestMatchers("/user/**").authenticated()
```

`/user/**` 경로는 로그인한 사용자라면 접근할 수 있다.

```text
/user/me 요청
→ 인증된 사용자 여부 확인
→ 인증되어 있으면 접근 허용
```

`/admin/**` 경로는 관리자 권한이 있어야 접근할 수 있다.

```text
/admin/members 요청
→ 인증 여부 확인
→ 사용자 권한 확인
→ ROLE_ADMIN이면 접근 허용
→ ROLE_USER이면 접근 거부
```

이 과정이 인가 처리이다.

### 2.7 Security Filter란?

Filter는 사용자의 요청이 Controller에 도착하기 전에 요청을 검사하는 구성 요소이다.

Spring Security는 여러 개의 보안 Filter를 연결하여 인증과 인가를 처리한다.

```text
사용자 요청
→ Security Filter들
→ Controller
```

토큰 인증을 사용하는 경우 Filter는 다음과 같은 일을 할 수 있다.

```text
Authorization 헤더 확인
→ 토큰 추출
→ 토큰 유효성 검증
→ 사용자 정보 조회
→ 인증 정보 등록
→ 다음 Filter로 요청 전달
```

토큰이 없거나 유효하지 않다면 Controller까지 요청을 전달하지 않고 차단할 수 있다.

### 2.8 SecurityFilterChain이란?

`SecurityFilterChain`은 Spring Security의 보안 규칙을 설정하는 객체이다.

어떤 API는 누구나 사용할 수 있고, 어떤 API는 로그인이 필요하며, 어떤 API는 관리자만 사용할 수 있는지 설정한다.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(
            HttpSecurity http
    ) throws Exception {

        http
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers(
                                "/auth/signup",
                                "/auth/login"
                        ).permitAll()

                        .requestMatchers(
                                "/admin/**"
                        ).hasRole("ADMIN")

                        .anyRequest().authenticated()
                );

        return http.build();
    }
}
```

위 코드를 해석하면 다음과 같다.

```text
/auth/signup
/auth/login
→ 로그인하지 않아도 접근 가능

/admin/**
→ 관리자 권한이 있는 사용자만 접근 가능

그 외 모든 요청
→ 로그인한 사용자만 접근 가능
```

주요 메서드는 다음과 같다.

| 메서드                | 의미                    |
| ------------------ | --------------------- |
| `permitAll()`      | 누구나 접근 가능             |
| `authenticated()`  | 인증된 사용자만 접근 가능        |
| `hasRole("ADMIN")` | 관리자 역할을 가진 사용자만 접근 가능 |
| `hasAnyRole()`     | 지정한 여러 역할 중 하나가 필요    |
| `denyAll()`        | 모든 사용자의 접근 거부         |

### 2.9 UserDetails란?

`UserDetails`는 Spring Security가 사용하는 사용자 정보 객체이다.

프로젝트에서는 일반적으로 `Member`와 같은 엔티티를 사용한다.

```text
Member
→ 우리 프로젝트에서 사용하는 회원 객체
```

하지만 Spring Security는 사용자의 아이디, 비밀번호, 권한 등을 일정한 형태로 다루기 위해 `UserDetails`를 사용한다.

```text
Member
→ Spring Security가 이해할 수 있는 UserDetails 형태로 변환
```

`UserDetails`에는 다음과 같은 정보가 포함될 수 있다.

- 사용자 아이디 또는 이메일
- 비밀번호
- 사용자 권한
- 계정 활성화 여부
- 계정 잠김 여부
- 계정 만료 여부

간단한 예시는 다음과 같다.

```java
public class CustomUserDetails implements UserDetails {

    private final Member member;

    public CustomUserDetails(Member member) {
        this.member = member;
    }

    @Override
    public String getUsername() {
        return member.getEmail();
    }

    @Override
    public String getPassword() {
        return member.getPassword();
    }

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return List.of(
                new SimpleGrantedAuthority(member.getRole().name())
        );
    }
}
```

### 2.10 UserDetailsService란?

`UserDetailsService`는 사용자의 정보를 데이터베이스에서 조회하는 역할을 한다.

사용자가 로그인을 시도하면 이메일이나 아이디를 이용하여 회원을 찾는다.

```java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService
        implements UserDetailsService {

    private final MemberRepository memberRepository;

    @Override
    public UserDetails loadUserByUsername(
            String email
    ) throws UsernameNotFoundException {

        Member member = memberRepository
                .findByEmail(email)
                .orElseThrow(
                        () -> new UsernameNotFoundException(
                                "사용자를 찾을 수 없습니다."
                        )
                );

        return new CustomUserDetails(member);
    }
}
```

처리 흐름은 다음과 같다.

```text
사용자가 이메일 입력
→ UserDetailsService가 이메일로 회원 조회
→ Member 객체 찾기
→ UserDetails 형태로 변환
→ Spring Security에 전달
```

### 2.11 PasswordEncoder란?

`PasswordEncoder`는 비밀번호를 암호화하고 비교하는 데 사용한다.

사용자의 비밀번호를 데이터베이스에 그대로 저장하면 안 된다.

잘못된 저장 예시는 다음과 같다.

```text
email: user@email.com
password: 12345678
```

데이터베이스가 노출되면 사용자의 실제 비밀번호도 함께 노출되기 때문이다.

따라서 회원가입 시 비밀번호를 암호화하여 저장한다.

Spring Security에서는 일반적으로 `BCryptPasswordEncoder`를 사용한다.

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

회원가입 시에는 `encode()`를 사용한다.

```java
String encodedPassword =
        passwordEncoder.encode(request.getPassword());
```

로그인 시에는 `matches()`를 사용한다.

```java
boolean matches = passwordEncoder.matches(
        request.getPassword(),
        member.getPassword()
);
```

두 값의 의미는 다음과 같다.

```text
request.getPassword()
→ 사용자가 로그인할 때 입력한 원본 비밀번호

member.getPassword()
→ 데이터베이스에 저장된 암호화된 비밀번호
```

### 2.12 인증 실패와 인가 실패

인증 실패와 인가 실패는 서로 다른 상황이다.

#### 인증 실패

인증 실패는 서버가 사용자가 누구인지 확인할 수 없는 경우이다.

예시는 다음과 같다.

- 로그인하지 않음
- 토큰이 없음
- 토큰이 만료됨
- 토큰이 잘못됨
- 세션이 만료됨
- 이메일 또는 비밀번호가 일치하지 않음

인증 실패에는 일반적으로 `401 Unauthorized`를 사용한다.

```text
401 Unauthorized
→ 너가 누군지 확인할 수 없다.
```

#### 인가 실패

인가 실패는 사용자가 누구인지는 확인되었지만 해당 기능을 사용할 권한이 없는 경우이다.

예시는 다음과 같다.

- 일반 사용자가 관리자 페이지에 접근
- 일반 사용자가 관리자 API 호출
- 다른 사용자의 게시글 삭제
- 권한이 없는 데이터에 접근

인가 실패에는 일반적으로 `403 Forbidden`을 사용한다.

```text
403 Forbidden
→ 너가 누군지는 알지만 이 기능을 사용할 권한이 없다.
```

비교하면 다음과 같다.

| 구분     | 인증 실패               | 인가 실패              |
| ------ | ------------------- | ------------------ |
| 상태 코드  | `401 Unauthorized`  | `403 Forbidden`    |
| 의미     | 사용자가 누구인지 확인할 수 없음  | 사용자는 확인했지만 권한이 없음  |
| 대표 예시  | 토큰 없이 API 요청        | 일반 사용자가 관리자 API 요청 |
| 로그인 상태 | 로그인되지 않았거나 인증 정보 오류 | 로그인된 상태            |

흐름으로 표현하면 다음과 같다.

```text
토큰 없이 /user/me 요청
→ 사용자가 누구인지 모름
→ 401 Unauthorized
```

```text
일반 사용자가 /admin/members 요청
→ 일반 사용자임을 확인
→ 관리자 권한 없음
→ 403 Forbidden
```

### 2.13 전체 핵심 정리

```text
인증(Authentication)
→ 사용자가 누구인지 확인하는 과정
→ 대표적인 예시는 로그인

인가(Authorization)
→ 인증된 사용자가 특정 기능을 사용할 권한이 있는지 확인하는 과정

인증과 인가의 순서
→ 인증이 인가보다 먼저 선행됨
→ 먼저 누구인지 확인하고, 그다음 권한을 확인함

세션
→ 서버가 로그인 상태를 저장
→ 사용자는 세션 ID를 전달

토큰
→ 사용자가 인증용 문자열을 보관
→ API 요청마다 토큰을 전달

Spring Security
→ 인증과 인가를 처리하는 Spring의 보안 프레임워크

Security Filter
→ Controller보다 먼저 사용자의 요청을 검사

SecurityFilterChain
→ URL별 접근 권한을 설정

UserDetails
→ Spring Security가 사용하는 사용자 정보

UserDetailsService
→ 데이터베이스에서 사용자 정보를 조회

PasswordEncoder
→ 비밀번호를 암호화하고 비교

401 Unauthorized
→ 사용자가 누구인지 확인할 수 없음

403 Forbidden
→ 사용자는 확인했지만 해당 기능을 사용할 권한이 없음
```

Spring Security의 전체 요청 흐름은 다음과 같다.

```text
사용자 API 요청
→ Spring Security Filter가 먼저 요청을 받음
→ 세션 또는 토큰 확인
→ 인증 정보 검증
→ 사용자가 누구인지 확인
→ 인증된 사용자로 등록
→ 해당 API에 필요한 권한 확인
→ 권한이 있으면 Controller 실행
→ 인증 실패 시 401
→ 인가 실패 시 403
```

## 3. Stateful과 Stateless

Stateful과 Stateless는 **서버가 사용자의 이전 상태를 기억하는지**에 따라 구분하는 방식이다.

여기서 상태란 다음과 같은 정보를 의미한다.

- 사용자가 로그인했는지
- 어떤 사용자로 로그인했는지
- 어떤 권한을 가지고 있는지
- 이전 요청과 현재 요청이 같은 사용자의 요청인지

쉽게 정리하면 다음과 같다.

```text
Stateful
→ 서버가 사용자의 로그인 상태를 기억한다.

Stateless
→ 서버가 사용자의 로그인 상태를 기억하지 않는다.
→ 사용자가 요청할 때마다 인증 정보를 전달한다.
```

### 3.1 상태(State)란?

웹에서 상태란 서버가 사용자의 이전 요청과 현재 요청을 연결하기 위해 필요한 정보이다.

예를 들어 사용자가 로그인한 후 자신의 정보를 조회한다고 가정한다.

```text
1. 사용자가 로그인한다.
2. 서버가 사용자를 인증한다.
3. 사용자가 내 정보 조회 API를 요청한다.
4. 서버가 현재 요청을 보낸 사용자가 누구인지 확인한다.
```

서버는 로그인 이후의 요청에서도 사용자를 구분해야 한다.

이때 사용자의 로그인 정보를 서버가 직접 저장하면 Stateful 방식이고, 사용자가 요청할 때마다 토큰과 같은 인증 정보를 보내면 Stateless 방식이다.

### 3.2 Stateful이란?

Stateful은 **서버가 사용자의 상태를 저장하고 기억하는 방식**이다.

대표적인 예시가 세션 기반 인증이다.

사용자가 로그인하면 서버는 사용자의 로그인 정보를 세션에 저장한다.

```text
사용자 로그인 성공
→ 서버가 세션 생성
→ 서버가 로그인 정보 저장
→ 사용자에게 세션 ID 전달
```

서버에는 다음과 같은 정보가 저장될 수 있다.

```text
세션 ID: ABC123
회원 ID: 1
이메일: user@email.com
권한: ROLE_USER
```

서버가 `ABC123`이라는 세션 ID와 사용자 정보를 연결해서 저장하고 있기 때문에, 이후 같은 세션 ID가 전달되면 해당 사용자를 확인할 수 있다.

```text
세션 ID ABC123 수신
→ 서버에 저장된 세션 조회
→ 회원 ID 1번 사용자 확인
→ ROLE_USER 권한 확인
```

즉, Stateful에서는 서버가 다음 내용을 기억하고 있다.

```text
“세션 ID가 ABC123인 사용자는 회원 ID 1번이다.”
```

### 3.3 Stateful의 처리 흐름

세션 기반 인증의 전체 흐름은 다음과 같다.

```text
1. 사용자가 이메일과 비밀번호로 로그인한다.

2. 서버가 사용자의 정보를 확인한다.

3. 이메일과 비밀번호가 일치하면 인증에 성공한다.

4. 서버가 세션을 생성한다.

5. 서버가 세션에 사용자 정보를 저장한다.

6. 서버가 사용자에게 세션 ID를 전달한다.

7. 브라우저가 세션 ID를 쿠키에 저장한다.

8. 사용자가 API를 요청할 때 브라우저가 세션 ID를 함께 전송한다.

9. 서버가 세션 ID를 이용하여 저장된 사용자 정보를 찾는다.

10. 서버가 사용자의 인증 상태와 권한을 확인한다.

11. 필요한 권한이 있으면 요청을 처리한다.
```

간단하게 표현하면 다음과 같다.

```text
로그인
→ 서버에 사용자 정보 저장
→ 세션 ID 발급
→ 브라우저가 세션 ID 보관
→ 이후 요청마다 세션 ID 전달
→ 서버가 세션을 조회하여 사용자 확인
```

### 3.4 쿠키란?

쿠키(Cookie)는 **브라우저가 저장하는 작은 데이터**이다.

서버가 브라우저에게 어떤 값을 저장하도록 전달하면, 브라우저는 그 값을 보관했다가 같은 서버에 다시 요청할 때 함께 전송할 수 있다.

세션 방식에서는 일반적으로 쿠키에 세션 ID를 저장한다.

```text
서버에 저장되는 것
→ 실제 로그인 정보가 들어 있는 세션

브라우저 쿠키에 저장되는 것
→ 세션을 찾기 위한 세션 ID
```

예를 들어 서버에는 다음 정보가 저장된다.

```text
세션 ID: ABC123
회원 ID: 1
이름: 김경민
권한: ROLE_USER
```

브라우저의 쿠키에는 세션 ID만 저장된다.

```text
JSESSIONID=ABC123
```

Spring Boot에서 세션을 사용할 때 세션 ID 쿠키 이름으로 `JSESSIONID`가 자주 사용된다.

이후 브라우저가 같은 서버에 API 요청을 보내면 다음과 같이 쿠키를 자동으로 함께 전송한다.

```http
Cookie: JSESSIONID=ABC123
```

서버는 전달받은 `ABC123`을 이용해 저장된 세션을 조회한다.

```text
JSESSIONID=ABC123 수신
→ ABC123 세션 조회
→ 회원 ID 1번 사용자 확인
→ 로그인된 사용자로 처리
```

### 3.5 세션과 쿠키의 차이

세션과 쿠키는 같은 것이 아니다.

```text
세션
→ 서버에 저장되는 사용자 로그인 정보

세션 ID
→ 서버에 저장된 세션을 찾기 위한 식별값

쿠키
→ 브라우저가 세션 ID를 보관하는 저장 공간
```

쉽게 비유하면 다음과 같다.

```text
세션
→ 식당이 보관하는 손님 정보

세션 ID
→ 손님에게 발급한 번호표 번호

쿠키
→ 사용자가 번호표를 넣어두는 지갑
```

사용자가 식당에 다시 방문할 때 번호표를 보여주면, 식당은 번호표를 보고 저장된 손님 정보를 찾을 수 있다.

### 3.6 쿠키를 사용하는 이유

HTTP는 기본적으로 이전 요청을 기억하지 않는다.

첫 번째 요청에서 로그인했다고 해도, 다음 요청만 보면 서버는 그 요청이 같은 사용자의 요청인지 바로 알 수 없다.

```text
첫 번째 요청
→ 로그인 성공

두 번째 요청
→ 서버 입장에서는 별도의 새로운 요청
```

따라서 브라우저가 요청할 때마다 세션 ID를 전송해야 한다.

이 작업을 쿠키가 자동으로 처리해준다.

```text
브라우저
→ 세션 ID를 쿠키에 저장
→ 같은 서버에 요청할 때 자동으로 세션 ID 전송

서버
→ 세션 ID를 이용하여 로그인 사용자 확인
```

### 3.7 세션 ID를 쿠키에 저장할 때 보안이 필요한 이유

세션 ID는 단순한 번호처럼 보이지만 실제로는 **로그인한 사용자를 증명하는 임시 출입증**과 같은 역할을 한다.

다른 사람이 세션 ID를 탈취하면 서버가 공격자를 원래 사용자로 잘못 판단할 수 있다.

```text
정상 사용자의 세션 ID: ABC123

공격자가 ABC123 탈취
→ 공격자가 ABC123을 서버에 전달
→ 서버가 정상 사용자로 착각
```

이렇게 세션 ID를 탈취하여 다른 사용자로 가장하는 공격을 **세션 하이재킹(Session Hijacking)**이라고 한다.

따라서 세션 ID를 저장하는 쿠키에는 보안 설정이 필요하다.

#### HttpOnly

`HttpOnly`는 브라우저의 JavaScript에서 쿠키에 접근하지 못하도록 제한한다.

```text
HttpOnly 적용
→ JavaScript에서 세션 쿠키를 읽기 어려움
```

예시는 다음과 같다.

```http
Set-Cookie: JSESSIONID=ABC123; HttpOnly
```

웹사이트에서 악성 JavaScript가 실행되더라도 세션 ID를 탈취하기 어렵게 만들어준다.

#### Secure

`Secure`는 HTTPS 연결에서만 쿠키를 전송하도록 설정한다.

```text
HTTP 요청
→ 쿠키 전송 제한

HTTPS 요청
→ 쿠키 전송
```

예시는 다음과 같다.

```http
Set-Cookie: JSESSIONID=ABC123; Secure
```

실제 배포 환경에서는 HTTPS를 사용하고 `Secure` 설정을 적용하는 것이 중요하다.

#### SameSite

`SameSite`는 다른 사이트에서 현재 사이트로 요청을 보낼 때 쿠키가 자동으로 전송되는 것을 제한한다.

주로 CSRF 공격을 방어하는 데 도움을 준다.

```http
Set-Cookie: JSESSIONID=ABC123; SameSite=Lax
```

대표적인 값은 다음과 같다.

| 설정       | 의미                                   |
| -------- | ------------------------------------ |
| `Strict` | 다른 사이트에서 시작한 요청에는 쿠키를 거의 보내지 않음      |
| `Lax`    | 일반적인 링크 이동은 허용하고 일부 위험한 요청은 제한       |
| `None`   | 다른 사이트 요청에도 쿠키 전송 가능하며 `Secure`가 필요함 |

#### 세션 만료 시간

세션을 너무 오래 유지하면 세션 ID가 유출되었을 때 공격자가 사용할 수 있는 시간도 길어진다.

```text
일정 시간 동안 요청이 없음
→ 세션 만료
→ 다시 로그인 필요
```

#### 로그인 성공 후 세션 ID 변경

로그인 전과 로그인 후에 같은 세션 ID를 계속 사용하면 세션 고정 공격에 이용될 수 있다.

따라서 로그인에 성공하면 새로운 세션 ID를 발급하는 것이 안전하다.

```text
로그인 전 세션 ID: AAA111
→ 로그인 성공
→ 로그인 후 세션 ID: BBB222
```

#### 로그아웃 시 세션 삭제

사용자가 로그아웃하면 서버에 저장된 세션을 삭제해야 한다.

```text
로그아웃 요청
→ 서버의 세션 삭제
→ 브라우저의 세션 쿠키 삭제
→ 기존 세션 ID 사용 불가
```

세션 쿠키의 주요 보안 설정을 정리하면 다음과 같다.

```text
HttpOnly
→ JavaScript의 쿠키 접근 제한

Secure
→ HTTPS에서만 쿠키 전송

SameSite
→ 다른 사이트 요청에서 쿠키 전송 제한

세션 만료
→ 일정 시간이 지나면 로그인 상태 종료

세션 ID 재발급
→ 세션 고정 공격 방지

로그아웃 시 세션 삭제
→ 기존 세션 ID 사용 차단
```

### 3.8 Stateful의 장점

#### 로그아웃 처리가 쉽다

사용자가 로그아웃하면 서버의 세션을 삭제하면 된다.

```text
사용자 로그아웃
→ 서버 세션 삭제
→ 기존 세션 ID 사용 불가
```

#### 로그인 상태를 서버에서 직접 관리할 수 있다

서버가 현재 로그인한 사용자의 정보를 가지고 있기 때문에 특정 사용자의 로그인 상태를 즉시 종료하기 쉽다.

#### 브라우저 기반 서비스에 사용하기 편리하다

브라우저가 쿠키를 자동으로 전송하기 때문에 사용자가 직접 세션 ID를 헤더에 추가하지 않아도 된다.

### 3.9 Stateful의 단점

#### 서버가 사용자별 세션을 저장해야 한다

로그인한 사용자가 많아질수록 서버가 저장해야 하는 세션도 많아진다.

```text
로그인 사용자 1명
→ 세션 1개 저장

로그인 사용자 10만 명
→ 많은 세션 저장 공간 필요
```

#### 서버를 여러 대 사용할 때 추가 구성이 필요하다

사용자가 서버 A에서 로그인했다고 가정한다.

```text
로그인 요청
→ 서버 A가 처리
→ 세션이 서버 A에 저장됨
```

다음 요청이 서버 B로 전달되면 서버 B에는 해당 세션이 없을 수 있다.

```text
다음 요청
→ 서버 B가 처리
→ 서버 B에는 세션 정보가 없음
→ 사용자를 확인하기 어려움
```

이를 해결하기 위해 Redis와 같은 공통 세션 저장소를 사용할 수 있다.

```text
서버 A ─┐
서버 B ─┼→ Redis와 같은 공통 세션 저장소
서버 C ─┘
```

Stateful 방식도 서버 확장이 가능하지만 세션을 공유하기 위한 추가 구성이 필요할 수 있다.

### 3.10 Stateless란?

Stateless는 **서버가 사용자의 로그인 상태를 세션으로 저장하지 않는 방식**이다.

대표적인 예시가 JWT 토큰 기반 인증이다.

사용자가 로그인에 성공하면 서버는 사용자에게 토큰을 발급한다.

```text
사용자 로그인 성공
→ 서버가 토큰 생성
→ 사용자에게 토큰 전달
→ 사용자가 토큰 보관
```

이후 사용자는 API를 요청할 때마다 토큰을 함께 보낸다.

```http
GET /user/me
Authorization: Bearer access-token
```

서버는 이전 로그인 상태를 세션에서 찾지 않는다.

현재 요청에 포함된 토큰을 검증하여 사용자가 누구인지 확인한다.

```text
토큰 수신
→ 토큰 유효성 검증
→ 사용자 정보 확인
→ 권한 확인
→ 요청 처리
```

즉, Stateless에서는 서버가 다음과 같이 동작한다.

```text
“내가 이전 로그인 상태를 저장해두지는 않았지만,
현재 요청에 들어 있는 토큰을 검증해서 사용자를 확인하겠다.”
```

### 3.11 Stateless의 처리 흐름

JWT 토큰 기반 인증의 전체 흐름은 다음과 같다.

```text
1. 사용자가 이메일과 비밀번호로 로그인한다.

2. 서버가 회원 정보를 조회한다.

3. 서버가 비밀번호를 확인한다.

4. 인증에 성공하면 서버가 토큰을 생성한다.

5. 서버가 사용자에게 토큰을 전달한다.

6. 사용자가 토큰을 보관한다.

7. 사용자가 API를 요청할 때 토큰을 함께 전송한다.

8. 서버가 토큰의 서명과 만료 시간을 검증한다.

9. 서버가 토큰에서 사용자 정보를 확인한다.

10. 서버가 사용자의 권한을 확인한다.

11. 필요한 권한이 있으면 요청을 처리한다.
```

간단히 표현하면 다음과 같다.

```text
로그인
→ 토큰 발급
→ 사용자가 토큰 보관
→ 요청마다 토큰 전달
→ 서버가 토큰 검증
→ 인증 및 인가 처리
```

### 3.12 JWT 토큰의 구조

JWT는 일반적으로 점(`.`)으로 구분된 세 부분으로 구성된다.

```text
Header.Payload.Signature
```

각 부분의 역할은 다음과 같다.

```text
Header
→ 어떤 알고리즘을 사용했는지 표시

Payload
→ 사용자 ID, 권한, 만료 시간 등의 정보

Signature
→ 토큰이 서버가 발급한 것이 맞는지 확인하는 서명
```

예를 들어 Payload에는 다음과 같은 정보가 들어갈 수 있다.

```json
{
  "sub": "1",
  "role": "ROLE_USER",
  "exp": 1780000000
}
```

각 값의 의미는 다음과 같다.

```text
sub
→ 사용자를 구분하는 값
→ 예: 회원 ID 1번

role
→ 사용자의 권한

exp
→ 토큰의 만료 시간
```

JWT의 Payload는 일반적으로 암호화된 정보가 아니라 인코딩된 정보이다.

따라서 비밀번호나 주민등록번호 같은 중요한 개인정보를 Payload에 넣으면 안 된다.

### 3.13 토큰은 어떻게 검증하는가?

토큰을 사용하려면 서버가 그 토큰이 자신이 발급한 정상적인 토큰인지 확인할 수 있어야 한다.

JWT에서는 주로 **서명(Signature)**을 이용해 검증한다.

서버가 토큰을 발급할 때 서버만 알고 있는 비밀키 또는 개인키를 이용해 서명을 만든다.

```text
Header + Payload + 서버의 비밀키
→ Signature 생성
```

사용자가 API 요청과 함께 토큰을 보내면 서버는 다음 순서로 토큰을 검증한다.

```text
1. 요청 헤더에서 토큰을 꺼낸다.

2. JWT의 구조가 정상인지 확인한다.

3. 서버가 가진 검증 키로 서명을 확인한다.

4. 토큰의 내용이 변경되지 않았는지 확인한다.

5. 토큰의 만료 시간이 지나지 않았는지 확인한다.

6. 필요한 경우 발급자와 사용 대상을 확인한다.

7. 모든 검증이 성공하면 인증된 사용자로 처리한다.
```

핵심은 다음과 같다.

```text
토큰의 서명이 정상임
→ 서버가 발급한 토큰이며 내용이 변조되지 않았다고 판단

토큰의 서명이 다름
→ 토큰이 변경되었거나 위조됨
→ 인증 실패
```

### 3.14 토큰의 Payload를 변경하면 어떻게 되는가?

JWT의 Payload는 확인할 수 있기 때문에 공격자가 다음과 같이 권한을 바꾸려고 할 수 있다.

```text
원래 권한
→ ROLE_USER

변경한 권한
→ ROLE_ADMIN
```

하지만 Payload를 변경하면 기존 Signature와 맞지 않게 된다.

```text
원래 Payload + 서버 비밀키
→ 원래 Signature

변경된 Payload + 기존 Signature
→ 서로 일치하지 않음
→ 검증 실패
```

공격자는 서버의 비밀키를 모르기 때문에 변경된 Payload에 맞는 정상적인 Signature를 만들기 어렵다.

따라서 서버는 토큰이 변조되었다는 사실을 확인할 수 있다.

### 3.15 토큰을 검증할 때 확인하는 항목

서버는 일반적으로 다음 내용을 확인한다.

#### 서명

서버가 신뢰하는 키로 발급된 토큰인지 확인한다.

```text
서명 정상
→ 검증 통과

서명 불일치
→ 위조되거나 변조된 토큰
```

#### 만료 시간

토큰이 아직 사용 가능한지 확인한다.

```text
현재 시간 < 토큰 만료 시간
→ 사용 가능

현재 시간 > 토큰 만료 시간
→ 만료된 토큰
→ 인증 실패
```

#### 발급자

`iss` 값을 이용하여 어떤 서버가 토큰을 발급했는지 확인할 수 있다.

```json
{
  "iss": "wrap-api"
}
```

#### 사용 대상

`aud` 값을 이용하여 어떤 서비스에서 사용할 토큰인지 확인할 수 있다.

```json
{
  "aud": "wrap-client"
}
```

#### 사용자 상태

토큰의 서명이 정상이어도 필요에 따라 데이터베이스에서 사용자를 추가로 확인할 수 있다.

```text
토큰에서 회원 ID 확인
→ DB에서 회원 조회
→ 탈퇴한 회원인지 확인
→ 정지된 계정인지 확인
→ 권한이 변경되었는지 확인
```

즉, 토큰 검증이 성공했다는 것은 기본적으로 다음을 의미한다.

```text
신뢰하는 서버가 발급한 토큰이다.
토큰의 내용이 변조되지 않았다.
토큰의 유효 기간이 지나지 않았다.
```

하지만 사용자가 현재 탈퇴 또는 정지 상태인지까지 항상 보장하는 것은 아니므로 서비스에 따라 DB 조회가 추가될 수 있다.

### 3.16 Spring Security에서 토큰을 검증하는 흐름

Spring Security와 JWT를 함께 사용하면 일반적으로 JWT Filter에서 토큰을 검증한다.

```text
사용자 요청
→ JWT Filter
→ Authorization 헤더 확인
→ 토큰 추출
→ 토큰의 서명과 만료 시간 검증
→ 사용자 ID와 권한 추출
→ Authentication 객체 생성
→ Spring Security에 인증 정보 등록
→ 권한 확인
→ Controller 실행
```

간단한 코드 형태는 다음과 같다.

```java
String token = getTokenFromRequest(request);

if (jwtProvider.validateToken(token)) {

    Authentication authentication =
            jwtProvider.getAuthentication(token);

    SecurityContextHolder
            .getContext()
            .setAuthentication(authentication);
}
```

각 코드의 의미는 다음과 같다.

```text
validateToken(token)
→ 토큰의 서명, 만료 시간 등을 검사

getAuthentication(token)
→ 토큰에서 사용자와 권한 정보 생성

setAuthentication(authentication)
→ 현재 요청을 인증된 요청으로 등록
```

토큰 검증에 성공해야 Spring Security가 현재 요청을 인증된 사용자의 요청으로 처리한다.

### 3.17 토큰을 검증할 수단만 있으면 되는가?

Stateless JWT 방식에서는 서버가 사용자마다 세션을 저장하지 않아도 된다.

각 서버가 토큰을 검증할 수 있는 키와 검증 로직을 가지고 있다면 요청을 처리할 수 있다.

```text
서버가 가지고 있어야 하는 것
→ 토큰 검증 키
→ 토큰 검증 로직
→ 만료 시간과 권한을 확인하는 로직
```

예를 들어 서버가 여러 대라면 각 서버가 같은 검증 기준을 사용할 수 있다.

```text
서버 A
→ 토큰 검증 가능

서버 B
→ 토큰 검증 가능

서버 C
→ 토큰 검증 가능
```

따라서 요청이 어떤 서버로 전달되더라도 세션 저장소를 조회하지 않고 토큰을 검증할 수 있다.

다만 비밀키를 여러 서버에 무분별하게 공유하는 것은 위험할 수 있다.

공개키 방식에서는 토큰을 발급하는 서버만 개인키를 가지고, 토큰을 검증하는 서버들은 공개키를 가지는 구조도 사용할 수 있다.

### 3.18 Stateless는 서버가 아무것도 저장하지 않는다는 뜻인가?

Stateless라고 해서 서버가 모든 정보를 저장하지 않는 것은 아니다.

서버는 여전히 다음 데이터를 저장한다.

- 회원 정보
- 게시글 정보
- 주문 정보
- 상품 정보
- 사용자 권한
- 서비스에서 필요한 데이터

Stateless에서 일반적으로 저장하지 않는 것은 **사용자별 로그인 세션 상태**이다.

```text
서버가 저장하는 것
→ 회원, 게시글, 상품 등의 서비스 데이터

서버가 세션으로 저장하지 않는 것
→ 이 사용자가 현재 로그인되어 있다는 사용자별 세션 정보
```

따라서 Stateless의 정확한 의미는 다음과 같다.

> 서버가 요청 사이에서 사용자별 로그인 세션 상태를 기억하지 않고, 매 요청에 포함된 인증 정보를 이용하여 사용자를 확인하는 방식이다.

### 3.19 토큰 방식은 세션 방식보다 비용이 적은가?

토큰 방식은 서버가 사용자별 세션을 저장하지 않기 때문에 **세션 저장 공간과 세션 관리 부담을 줄일 수 있다.**

```text
세션 방식
→ 로그인 사용자마다 세션 정보 저장
→ 요청마다 세션 저장소 조회

토큰 방식
→ 사용자별 세션을 저장하지 않음
→ 요청마다 토큰 검증
```

로그인한 사용자가 많아질수록 세션 방식에서는 저장해야 하는 세션 수도 증가한다.

```text
로그인 사용자 1만 명
→ 최대 1만 개의 세션 관리 필요
```

JWT 방식에서는 사용자별 세션을 저장하지 않으므로 세션 저장 비용을 줄일 수 있다.

그러나 토큰 방식이 무조건 모든 비용이 적은 것은 아니다.

토큰 방식에서는 다른 비용이 발생한다.

#### 토큰 검증 연산

요청이 들어올 때마다 JWT의 서명을 확인해야 한다.

```text
모든 인증 요청
→ JWT 해석
→ 서명 검증
→ 만료 시간 확인
```

이 과정에는 CPU 연산이 필요하다.

#### 요청 크기 증가

세션 ID는 일반적으로 짧은 문자열이지만 JWT는 사용자 정보와 서명이 포함되어 있어 상대적으로 길다.

```text
세션 ID
→ 비교적 짧음

JWT
→ Header, Payload, Signature가 포함되어 상대적으로 김
```

JWT를 모든 요청에 포함하면 네트워크로 전송되는 데이터가 조금 증가한다.

#### 로그아웃과 강제 만료 처리

세션 방식은 서버의 세션을 삭제하면 즉시 로그아웃된다.

```text
세션 삭제
→ 즉시 사용 불가
```

JWT Access Token은 한 번 발급되면 일반적으로 만료 시간까지 자체적으로 유효하다.

```text
로그아웃
→ 클라이언트의 토큰 삭제

하지만 탈취된 토큰이 존재한다면
→ 토큰 만료 전까지 사용될 가능성이 있음
```

이를 보완하기 위해 다음 방법을 사용할 수 있다.

- Access Token의 만료 시간을 짧게 설정
- Refresh Token 사용
- Refresh Token을 DB나 Redis에 저장
- 로그아웃된 토큰을 차단 목록에 등록
- 계정 상태를 DB에서 추가 확인

이런 기능을 사용하면 토큰 방식에서도 DB나 Redis와 같은 저장소가 필요할 수 있다.

따라서 정확한 표현은 다음과 같다.

> 토큰 방식은 사용자별 세션 저장 비용을 줄이고 서버 확장을 쉽게 만들지만, 토큰 검증 연산과 로그아웃·만료 관리 비용이 추가된다.

### 3.20 Stateful과 Stateless의 비용 비교

| 구분        | Stateful           | Stateless                 |
| --------- | ------------------ | ------------------------- |
| 로그인 상태 저장 | 서버가 사용자별 세션 저장     | 일반적으로 사용자별 세션을 저장하지 않음    |
| 요청 처리 방식  | 세션 저장소 조회          | 토큰 서명 및 만료 시간 검증          |
| 저장 공간     | 로그인 사용자 수에 따라 증가   | 세션 저장 공간은 상대적으로 적음        |
| CPU 연산    | 세션 조회 중심           | 요청마다 토큰 검증 연산             |
| 요청 데이터 크기 | 세션 ID가 비교적 짧음      | JWT가 상대적으로 김              |
| 서버 확장     | 세션 공유 구성이 필요할 수 있음 | 각 서버가 토큰을 검증할 수 있어 비교적 쉬움 |
| 로그아웃      | 세션 삭제로 즉시 처리 가능    | 만료 및 차단 전략이 필요할 수 있음      |

토큰 방식은 무조건 비용이 적은 방식이라기보다 **비용이 발생하는 위치가 달라지는 방식**이라고 이해하는 것이 정확하다.

```text
Stateful
→ 세션을 저장하고 조회하는 비용

Stateless
→ 토큰을 전달하고 검증하는 비용
```

### 3.21 Stateless의 장점

#### 서버 확장이 비교적 쉽다

각 서버가 토큰을 검증할 수 있다면 요청이 어느 서버에 전달되어도 처리할 수 있다.

```text
첫 번째 요청
→ 서버 A가 토큰 검증

두 번째 요청
→ 서버 B가 토큰 검증

세 번째 요청
→ 서버 C가 토큰 검증
```

#### 프론트엔드와 백엔드가 분리된 구조에 적합하다

React, Next.js 또는 모바일 앱에서 Spring Boot API 서버를 호출하는 구조에 적합하다.

```text
React 또는 모바일 앱
→ 토큰을 포함한 API 요청
→ Spring Boot 서버
```

#### 다양한 클라이언트에서 사용할 수 있다

브라우저뿐 아니라 모바일 앱과 다른 서버에서도 HTTP 헤더에 토큰을 포함하여 요청할 수 있다.

### 3.22 Stateless의 단점

#### 토큰 탈취 위험

토큰이 탈취되면 만료되기 전까지 공격자가 사용할 가능성이 있다.

따라서 토큰을 안전하게 저장하고 HTTPS를 사용해야 한다.

#### 로그아웃 처리가 복잡할 수 있다

서버가 토큰 상태를 기억하지 않으면 발급된 Access Token을 즉시 무효화하기 어렵다.

#### 권한 변경이 즉시 반영되지 않을 수 있다

토큰에 `ROLE_USER` 권한이 들어 있는 상태에서 사용자의 권한을 변경해도 기존 토큰에는 이전 권한이 남아 있을 수 있다.

```text
토큰 발급 당시
→ ROLE_USER

DB에서 권한 변경
→ ROLE_ADMIN

기존 토큰
→ 아직 ROLE_USER 정보 포함
```

이 문제를 해결하려면 토큰 유효 시간을 짧게 설정하거나 요청 시 사용자 정보를 다시 조회하는 방법을 사용할 수 있다.

### 3.23 Stateful과 Stateless 비교

| 구분               | Stateful         | Stateless                  |
| ---------------- | ---------------- | -------------------------- |
| 상태 저장            | 서버가 로그인 상태 저장    | 서버가 로그인 세션 상태를 저장하지 않음     |
| 대표 인증 방식         | 세션 기반 인증         | JWT 토큰 기반 인증               |
| 사용자가 보관하는 것      | 세션 ID            | Access Token               |
| 서버가 사용자를 확인하는 방법 | 세션 ID로 저장된 세션 조회 | 토큰의 서명과 만료 시간 검증           |
| 브라우저 저장 방식       | 주로 쿠키            | 쿠키 또는 클라이언트 저장소            |
| 서버 확장            | 세션 공유가 필요할 수 있음  | 상대적으로 쉬움                   |
| 로그아웃             | 세션 삭제            | 토큰 만료 또는 차단 전략 필요          |
| 대표적인 사용 환경       | 브라우저 중심 웹 서비스    | REST API, 모바일 앱, 분리형 웹 서비스 |

쉽게 비교하면 다음과 같다.

```text
Stateful
→ 서버: “이 사용자를 내가 기억하고 있다.”

Stateless
→ 사용자: “내가 누구인지 증명하는 토큰을 가져왔다.”
```

### 3.24 상황별 사용 방식

#### Stateful이 잘 맞는 상황

- Spring Boot가 화면과 서버를 함께 담당하는 서비스
- 브라우저 중심 웹사이트
- 회사 내부 관리자 시스템
- 학교 홈페이지
- 서버에서 로그인 상태를 직접 관리하고 싶은 경우
- 로그아웃 또는 강제 로그아웃을 즉시 적용해야 하는 경우

```text
브라우저
↔ Spring Boot 서버
↔ 데이터베이스
```

#### Stateless가 잘 맞는 상황

- React와 Spring Boot가 분리된 서비스
- Next.js와 Spring Boot가 분리된 서비스
- 모바일 애플리케이션
- REST API 중심 서비스
- 여러 서버로 확장하는 서비스
- 프론트엔드와 백엔드를 각각 배포하는 서비스

```text
React 또는 모바일 앱
→ Spring Boot API 서버
→ 데이터베이스
```

예를 들어 다음과 같은 배포 구조에서는 JWT 토큰 방식이 많이 사용된다.

```text
프론트엔드
→ Vercel

백엔드
→ Render 또는 Railway

데이터베이스
→ Supabase
```

### 3.25 Spring Security와 Stateful

Spring Security는 세션 기반 Stateful 인증을 지원한다.

사용자가 로그인에 성공하면 인증 정보를 세션에 저장할 수 있다.

```text
로그인 성공
→ Authentication 객체 생성
→ 인증 정보를 서버 세션에 저장
→ 사용자에게 세션 ID 전달
```

이후 요청에서는 세션 ID를 이용하여 인증 정보를 확인한다.

```text
사용자 요청
→ 세션 ID 쿠키 확인
→ 서버 세션에서 Authentication 조회
→ 인증된 사용자 확인
→ 권한 검사
→ Controller 실행
```

### 3.26 Spring Security와 Stateless

Spring Security는 JWT와 함께 Stateless 방식으로도 사용할 수 있다.

이 경우 로그인 정보를 서버 세션에 저장하지 않고 요청마다 토큰을 검증한다.

```text
사용자 요청
→ JWT Filter가 토큰 추출
→ 토큰의 서명과 만료 시간 검증
→ 사용자 정보 확인
→ Authentication 객체 생성
→ Spring Security에 인증 정보 등록
→ 권한 검사
→ Controller 실행
```

Spring Security에서 세션을 생성하지 않도록 설정하는 예시는 다음과 같다.

```java
http
        .sessionManagement(session -> session
                .sessionCreationPolicy(
                        SessionCreationPolicy.STATELESS
                )
        );
```

이 설정은 다음과 같은 의미이다.

```text
Spring Security가 사용자 인증 상태를 세션에 저장하지 않음
→ 각 요청에 포함된 토큰으로 사용자를 확인
```

### 3.27 Stateful과 Stateless의 인증·인가 처리

Stateful과 Stateless는 사용자를 확인하는 방법은 다르지만 두 방식 모두 인증과 인가를 수행한다.

#### Stateful 방식

```text
세션 ID 확인
→ 서버에 저장된 세션 조회
→ 사용자 인증
→ 사용자 권한 확인
→ 인가 처리
→ Controller 실행
```

#### Stateless 방식

```text
토큰 확인
→ 서명과 만료 시간 검증
→ 사용자 인증
→ 사용자 권한 확인
→ 인가 처리
→ Controller 실행
```

두 방식의 가장 큰 차이는 다음과 같다.

```text
Stateful
→ 서버에 저장된 세션으로 사용자 확인

Stateless
→ 현재 요청에 포함된 토큰으로 사용자 확인
```

### 3.28 핵심 정리

```text
Stateful
→ 서버가 사용자의 로그인 상태를 기억하는 방식
→ 대표적인 예시는 세션 기반 인증
→ 사용자 정보는 서버의 세션에 저장
→ 브라우저는 세션 ID를 쿠키에 저장
→ 로그아웃과 강제 만료가 비교적 쉬움
→ 서버가 여러 대이면 세션 공유 구성이 필요할 수 있음
```

```text
쿠키
→ 브라우저가 저장하는 작은 데이터
→ 세션 방식에서는 세션 ID를 저장하는 데 사용

세션 ID 보안
→ HttpOnly, Secure, SameSite 설정 필요
→ 로그아웃 시 세션 삭제 필요
→ 세션 ID가 탈취되면 다른 사람이 사용자인 척할 수 있음
```

```text
Stateless
→ 서버가 사용자별 로그인 세션 상태를 기억하지 않는 방식
→ 대표적인 예시는 JWT 토큰 기반 인증
→ 사용자가 요청마다 토큰을 전달
→ 서버는 서명과 만료 시간을 검증
→ 여러 서버로 확장하기 상대적으로 쉬움
```

```text
토큰 검증
→ 토큰의 서명 확인
→ 만료 시간 확인
→ 발급자와 사용 대상 확인 가능
→ 검증에 성공하면 인증된 사용자로 처리
```

```text
토큰 방식의 비용
→ 사용자별 세션 저장 비용은 줄어듦
→ 요청마다 토큰 검증 연산이 필요함
→ JWT로 인해 요청 크기가 커질 수 있음
→ 로그아웃과 강제 만료 처리가 복잡할 수 있음
```

가장 쉽게 구분하면 다음과 같다.

```text
Stateful
→ 서버가 나를 기억한다.

Stateless
→ 내가 요청할 때마다 나를 증명한다.
```

따라서 토큰 방식이 무조건 비용이 적거나 세션 방식보다 항상 좋은 것은 아니다.

> Stateful은 서버가 세션을 저장하고 관리하는 비용이 발생하고, Stateless는 토큰을 전달하고 검증하며 만료와 로그아웃을 관리하는 비용이 발생한다.

서비스의 구조, 사용자 수, 서버 확장 여부, 로그아웃 처리 방법 등을 고려하여 적절한 방식을 선택해야 한다.

# Spring Security 추가 후 Swagger에 접속할 수 없었던 문제

## 1. 발생한 상황

프로젝트에 다음 Spring Security 의존성을 추가했다.

```gradle
implementation 'org.springframework.boot:spring-boot-starter-security'
```

의존성을 추가하고 애플리케이션을 다시 실행한 뒤 Swagger에 접속했다.

```text
http://localhost:8080/swagger-ui.html
```

하지만 Swagger 화면이 열리지 않고 Spring Security의 기본 로그인 화면으로 이동했다.

```text
Please sign in
Username
Password
```

회원가입 API와 H2 콘솔도 마찬가지로 접근할 수 없었다.

## 2. 원인

Spring Security는 의존성만 추가해도 기본적으로 애플리케이션의 모든 요청을 인증 대상으로 설정한다.

```text
/members/signup   → 인증 필요
/members/login    → 인증 필요
/swagger-ui.html  → 인증 필요
/h2-console       → 인증 필요
기타 API           → 인증 필요
```

별도의 `SecurityFilterChain` 설정이 없었기 때문에 Swagger 요청도 보호 대상이 됐다.

처리 과정은 다음과 같았다.

```text
Swagger 접속
→ Spring Security가 요청 확인
→ 로그인하지 않은 사용자로 판단
→ 접근 차단
→ 기본 로그인 화면으로 이동
```

이는 Swagger나 서버의 오류가 아니라 Spring Security의 기본 보안 설정이 정상적으로 작동한 결과였다.

## 3. 바로 Security를 추가하지 않은 이유

회원가입 기능을 먼저 완성하고 Security를 나중에 추가했다.

먼저 확인한 기능:

```text
회원가입 DTO 검증
→ 이메일 중복 검사
→ BCrypt 비밀번호 암호화
→ 회원 DB 저장
→ 예외 응답 확인
```

그다음 추가한 기능:

```text
Spring Security
→ DB 회원 조회
→ 로그인 연결
→ 공개·비공개 URL 설정
→ 401·403 처리
```

Security를 처음부터 추가하면 다음 문제를 동시에 확인해야 한다.

- DTO 검증 문제
- DB 저장 문제
- BCrypt 암호화 문제
- Security 접근 차단 문제
- 로그인 설정 문제
- 401·403 응답 문제

단계별로 구현하면 오류가 회원가입 로직에서 발생했는지 Security 설정에서 발생했는지 쉽게 구분할 수 있다.

## 4. 해결 방법

`SecurityConfig`에 `SecurityFilterChain`을 등록하고 공개할 주소를 직접 지정했다.

```java
@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(
            HttpSecurity http
    ) throws Exception {

        http
                .csrf(csrf -> csrf.disable())

                .authorizeHttpRequests(auth -> auth
                        .requestMatchers(
                                "/members/signup",
                                "/members/login",
                                "/swagger-ui.html",
                                "/swagger-ui/**",
                                "/v3/api-docs/**",
                                "/error"
                        ).permitAll()

                        .requestMatchers(
                                PathRequest.toH2Console()
                        ).permitAll()

                        .anyRequest().authenticated()
                )

                .formLogin(form -> form
                        .loginProcessingUrl("/members/login")
                        .usernameParameter("email")
                        .passwordParameter("password")
                        .permitAll()
                )

                .headers(headers -> headers
                        .frameOptions(frame -> frame.sameOrigin())
                );

        return http.build();
    }
}
```

## 5. 주요 설정의 의미

### 공개 API

```java
.requestMatchers(
        "/members/signup",
        "/members/login"
).permitAll()
```

회원가입과 로그인은 인증받기 전에도 사용할 수 있어야 하므로 공개한다.

### Swagger 공개

```java
.requestMatchers(
        "/swagger-ui.html",
        "/swagger-ui/**",
        "/v3/api-docs/**"
).permitAll()
```

각 주소의 역할은 다음과 같다.

```text
/swagger-ui.html → Swagger 최초 접속 주소
/swagger-ui/**   → Swagger의 HTML, JavaScript, CSS 파일
/v3/api-docs/**  → Swagger가 읽는 API 명세
```

일부 주소가 빠지면 Swagger 화면이 열리지 않거나 API 목록을 불러오지 못할 수 있다.

### 나머지 API 보호

```java
.anyRequest().authenticated()
```

공개 목록에 없는 API는 로그인한 사용자만 사용할 수 있다.

## 6. 로그인 ID가 이메일인 이유

폼 로그인 설정에서 다음과 같이 지정했다.

```java
.usernameParameter("email")
```

따라서 기본 로그인 화면에는 `Username`이라고 표시되더라도 실제로 입력할 값은 회원 이메일이다.

```text
Username: test@example.com
Password: password123
```

Spring Security는 입력된 이메일을 다음 메서드로 전달한다.

```java
loadUserByUsername(String email)
```

그 후 DB에서 회원을 조회한다.

```java
memberRepository.findByEmail(email)
```

## 7. 정상 동작 확인

설정 적용 후 다음 내용을 확인했다.

```text
Swagger 로그인 없이 접속 성공
회원가입 로그인 없이 성공
DB에 BCrypt 비밀번호 저장
이메일로 회원 조회
로그인 성공
인증 세션 생성
```

로그인 로그에는 다음과 같은 회원 조회 쿼리가 출력됐다.

```sql
select ...
from members
where email=?
```

이는 `MemberUserDetailsService`가 이메일로 DB 회원을 조회했다는 뜻이다.

## 8. 같은 문제가 생겼을 때 확인할 체크리스트

Spring Security 추가 후 특정 화면이나 API가 로그인 페이지로 이동하면 다음을 확인한다.

1. `SecurityConfig`가 실제로 생성되어 있는가?
2. 클래스에 `@Configuration`이 붙어 있는가?
3. `SecurityFilterChain`이 `@Bean`으로 등록되어 있는가?
4. 공개할 주소가 `requestMatchers(...).permitAll()`에 포함되어 있는가?
5. Swagger라면 세 경로가 모두 공개되어 있는가?

```text
/swagger-ui.html
/swagger-ui/**
/v3/api-docs/**
```

6. 회원가입과 로그인 주소가 공개되어 있는가?

```text
/members/signup
/members/login
```

7. 파일 저장 후 애플리케이션을 완전히 재시작했는가?
8. 주소 오타나 패키지 import 오류가 없는가?
9. 공개하지 않은 API가 로그인 화면으로 이동한다면 의도된 동작인가?

## 9. 핵심 정리

```text
Spring Security 의존성만 추가
→ 모든 요청이 기본적으로 보호됨
→ Swagger와 회원가입도 로그인 화면으로 이동

SecurityConfig 추가
→ 공개할 URL을 permitAll로 지정
→ 나머지는 authenticated로 보호
```

이번 문제는 Spring Security 오류가 아니라 공개 경로를 아직 설정하지 않아 발생한 정상적인 기본 동작이었다.
