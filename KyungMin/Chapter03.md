## 핵심 키워드 정리

## 1. SOLID 원칙이란?

### 1. SRP (Single Responsibility Principle) : 단일 책임 원칙

하나의 클래스는 하나의 책임(역할)만 가져야 한다.

쉽게 말해 하나의 클래스가 비즈니스 로직, DB 접근 등등의 역할을 하면 안 된다.

---

### 2. OCP (Open-Closed Principle) : 개방-폐쇄 원칙

확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다.

기존 코드를 변경하지 않으면서 기능을 확장할 수 있어야 한다.

**스프링 예시**

인터페이스를 활용하는 것이다.

결제 시스템을 만들면 `paymentService` 인터페이스를 두고 네이버페이, 카카오페이 클래스를 구현한다.

그런데 나중에 토스페이를 추가하고 싶을 때 기존 결제 핵심 로직 코드는 손대지 않고, 새로운 구현 클래스만 추가하면 된다.

- 기존 코드 수정 없음 → 수정에는 닫힘
- 새로운 기능 추가 가능 → 확장에는 열림

---

### 3. LSP (Liskov Substitution Principle) : 리스코프 치환 원칙

서브 타입은 언제나 자신의 기반 타입, 즉 부모 타입으로 교체할 수 있어야 한다.

다형성을 지키기 위한 규칙이다.

부모 클래스나 인터페이스 자리에 자식 클래스를 넣어도 프로그램이 의도한 대로 정확하게 작동해야 한다는 원칙이다.

단순히 컴파일이 되는 것을 넘어, 인터페이스의 규약을 자식이 마음대로 깨뜨리면 안 된다.

**잘못된 예시**

자동차 인터페이스의 `액셀을 밟다()`라는 기능은 앞으로 전진해야 한다.

그런데 이를 구현한 `페라리` 클래스가 속도를 줄이거나 후진하게 만들었으면 LSP 위반이다.

---

### 5. DIP (Dependency Inversion Principle) : 의존관계 역전 원칙

구체적인 클래스에 의존하지 말고, 인터페이스(역할)에 의존해야 한다.

변하기 쉬운 것, 즉 구현체 클래스보다 변하지 않는 것, 즉 인터페이스에 의존하라는 원칙이다.

이 원칙이 있어야 OCP, 즉 개방-폐쇄 원칙도 가능해진다.

**스프링 부트에서의 예시**

스프링의 핵심 중의 핵심 기능인 DI(Dependency Injection, 의존성 주입)가 바로 이 DIP를 실현해 주는 기술이다.

---

## 2. DI(Dependency Injection, 의존성 주입)

### 의존성이란?

의존성은 한 클래스가 작동하기 위해 다른 클래스를 필요로 하는 상태이다.

즉, 한 객체가 다른 객체를 사용할 때 의존성이 있다고 한다.

두 객체 간의 관계, 즉 의존성을 맺어주는 것을 의존성 주입이라고 한다.

---

### 의존성 주입 방법

의존성 주입에는 다음과 같은 방법이 있다.

- 생성자 주입
- 필드 주입
- 수정자 주입

하지만 Spring 4부터 생성자 주입을 강력히 권장하고 있다.

---

### 의존성 주입이 필요한 이유

두 클래스가 강하게 결합되어 있는 경우, 강하게 결합된 것이 문제가 된다.

생성자 변경이 필요한 경우 코드를 힘들게 수정해야 하므로 유연성이 떨어진다.

---

### DI의 세 가지 조건

DI의 세 가지 조건이 있다고 한다.  
단, 이것이 공식적인 정의는 아니다.

- 클래스 모델이나 코드에는 런타임 시점의 의존관계가 드러나지 않는다. 그러기 위해서는 인터페이스만 의존하고 있어야 한다.
- 런타임 시점의 의존관계는 컨테이너나 팩토리 같은 제3의 존재가 결정한다.
- 의존관계는 사용할 오브젝트에 대한 레퍼런스를 외부에서 제공, 즉 주입해줌으로써 만들어진다.

---

### 1. 강한 의존관계

```java
class A {
    private final B b;

    public A() {
       b = new b();
    }
}
```

이 코드는 `A`가 변하면 `B`에 영향이 갈 수 있다.

---

### 2. 추상화를 이용하여 결합도를 낮추기

```java
class A {
    private final BInterface bInterface;

    public A() {
        bInteface = new BInterfaceImpl1();
        //bInteface = new BInterfaceImpl2();
        //bInteface = new BInterfaceImpl3();
    }
    
    public void print() {
        System.out.println(bInterface.print());
    }
}

interface BInterface {
    print();
}

class BInterfaceImpl1 implements BInterface {

    @Override
    public String print() {
        return "B1";
    }
}
```

추상화를 이용하여 관계를 조금 느슨하게 하여 결합도를 낮춘다.

하지만 아직까지도 추상화된 Interface인 `BInterface`뿐 아니라 구체 클래스인 `BInterfaceImpl1`, `BInterfaceImpl2`, `BInterfaceImpl3`도 의존하고 있다.

---

### 3. 외부에서 DI 주입하기

```java
class A {
    private final BInterface bInterface;

    public A(BInterface bInterface) {
        this.bInterface = bInterface;
    }
    
    public void print() {
        System.out.println(bInterface.print());
    }
}

A a = new A(new BInterfaceImpl1());
a.print();    //B1
```

외부에서 DI를 주입해서 DIP까지 만족시킨다.

하지만 OCP를 만족하지 못하였다.

---

### 4. 스프링을 이용해서 DI 주입하기

```java
@Configuration
public class AppConfig {

    @Bean
    public BInterface bInterface() {
        return new BInterfaceImpl1(); 
        // 이 부분만 수정하면 언제든 다른 것으로 변경 가능
    }
}

AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
A a = ac.getBean(A.class);    //b = BInterfaceImpl1
a.print();    //B1
```

스프링을 이용해서 DI 주입 시 문제를 해결할 수 있다.

Spring 컨테이너가 Bean을 생성할 때 알아서 의존성을 찾아서 자동으로 주입한다.

---

### Bean 등록

`@Component`, `@Service`, `@Repository` 등을 기반으로 Bean을 등록한다.

사실 `@Service`와 `@Repository` 안에 `@Component` 어노테이션이 있다.

---

### DI 장점

- **결합도의 감소**  
  DI를 사용하면 객체 간의 결합도가 낮아지므로 개발 유지보수성이 높아진다.

- **테스트 용이성**  
  DI를 사용하면 의존성을 주입하기 때문에 객체의 동작을 검증하기 위해 테스트 객체를 주입하여 테스트 용이성을 높일 수 있다.

- **재사용성**  
  객체 간의 결합도를 낮추기 때문에 객체의 재사용성을 높인다.

- **가독성**  
  DI는 객체 간 의존성을 명시하고 객체 생성 및 의존성 주입을 한 곳에서 관리하기 때문에 코드의 가독성을 높일 수 있다.

---

## 3. IoC란?

IoC(Inversion of Control)는 **제어의 역전**을 의미한다.

메소드나 객체의 호출 작업을 개발자가 결정하는 것이 아니라 외부에서 결정되는 것을 의미한다.

제어의 역전이라는 개념은 전통적인 제어 흐름을 뒤집는다는 의미를 담고 있다.

전통적인 방식에서는 객체가 스스로 필요한 의존성을 직접 생성하거나 호출하는 방식이다.

하지만 IoC에서는 이 흐름이 역전된다.

객체 스스로가 의존성을 관리하지 않고 외부에서 그 제어를 맡는다.

이렇게 함으로써 객체 간의 강한 결합을 느슨한 결합으로 전환할 수 있다.

---

### IoC 예시

객체 `A`는 객체 `B`가 필요하지만, `B` 객체를 직접 생성하지 않고 외부에서 `B` 객체를 주입받게 된다.

이 외부의 주입자는 프레임워크나 IoC 컨테이너일 수 있다.

IoC가 적용됨으로써 제어권이 객체에서 외부로 역전된 것이다.

---

### IoC가 필요한 이유

객체 간의 결합도를 낮추기 위함이다.

전통적인 객체 생성 방식에서는 클래스가 스스로 의존성을 관리하며, 이는 코드가 매우 강하게 결합되는 것을 의미한다.

이러한 강한 결합은 유지보수를 어렵게 하고, 코드의 변경이 발생할 때마다 여러 부분을 수정해야 하는 문제를 야기할 수 있다.

큰 규모의 애플리케이션에서 의존성 관리를 일일이 수동으로 하게 되면 복잡성이 매우 높아진다.

IoC는 복잡성을 해결해주는 데 유용하다.

IoC는 객체의 생성 및 관리 책임을 외부로 넘겨, 재사용성, 유연성 그리고 테스트 가능성을 높인다.

---

### IoC의 장점

- **결합도 감소**  
  IoC는 객체 간의 결합도를 낮춰 코드 변경에 유연하게 대응할 수 있게 해준다. 이를 통해 유지보수성이 향상되고, 시스템의 확장성이 높아진다.

- **유연성 증가**  
  객체가 필요한 의존성을 외부에서 주입받으므로, 다양한 객체를 쉽게 교체할 수 있어 애플리케이션의 유연성이 증가한다.

- **재사용성**  
  IoC를 통해 코드를 재사용하기 쉬워진다. 의존성이 외부에서 관리되어, 동일한 코드가 다양한 환경에서 사용될 수 있다.

- **테스트 용이성**  
  IoC는 단위 테스트를 더 쉽게 만들어준다. 의존성을 외부에서 주입받으므로, 테스트할 때 모의 객체를 사용해 각 구성 요소를 독립적으로 테스트할 수 있다.

- **확장성**  
  IoC는 객체 간의 관계를 외부에서 정의하고 관리해서, 새로운 기능을 추가하거나 시스템을 확장하는 작업이 쉬워진다.

---

## 4. 생성자 주입 vs 수정자 주입 vs 필드 주입

### 생성자 주입

생성자 주입은 객체를 생성하는 시점에 생성자를 통해 의존 관계를 주입하는 방식이다.

강력히 권장하는 방식이다.

생성자 주입은 생성자의 호출 시점에 1회 호출되는 것이 보장된다.

그렇기 때문에 주입받은 객체가 변하지 않거나, 반드시 객체의 주입이 필요한 경우에 강제하기 위해 사용할 수 있다.

---

### 필드 주입

필드 주입은 일단 객체를 만든 후 의존성을 주입한다.

필드 주입을 이용하면 코드가 간결해지기 때문에 과거에 많이 사용되었던 주입 방법이다.

하지만 필드 주입은 외부에서 접근이 불가능하다는 단점이 존재한다.

테스트 코드의 중요성이 부각됨에 따라 필드의 객체를 수정할 수 없는 필드 주입은 거의 사용되지 않게 되었다.

필드 주입은 반드시 DI 프레임워크가 존재해야 하므로 반드시 사용을 지양해야 한다.

---

### 수정자 주입

수정자 주입은 필드 값을 변경하는 Setter를 통해서 의존 관계를 주입하는 방법이다.

수정자 주입은 생성자 주입과 다르게 주입받는 객체가 변경될 가능성이 있는 경우에 사용한다.

실제로 변경이 필요한 경우는 극히 드물다고 한다.

---

### 정리

생성자 주입은 객체의 불변성을 보장 가능하고 테스트 용이성이 높다.

하지만 수정자 주입이나 필드 주입은 객체의 불변성과 테스트 용이성이 떨어지기 때문에 현재는 생성자 주입을 사용하고 있다.

---

## 5. AOP란?

AOP(Aspect Oriented Programming)는 **관점 지향 프로그래밍**이다.

어떤 로직을 기준으로 핵심적인 관점, 부가적인 관점으로 나누어서 보고 그 관점을 기준으로 각각 모듈화하겠다는 것이다.

쉽게 말해서 중복되는 부가 기능 코드를 한 곳에 모아 관리하는 기술이라고 생각하면 된다.

---

### AOP 예시

모든 메소드의 호출 시간을 측정하고 싶은 상황이 있다고 하자.

예를 들어 회원 가입 시간, 회원 조회 시간을 측정하고 싶은 경우이다.

이러한 경우 문제는 다음과 같다.

- 회원가입, 회원 조회에 시간을 측정하는 기능은 핵심 관심 사항이 아니다.
- 시간을 측정하는 로직은 공통 관심 사항이다.
- 시간을 측정하는 로직과 핵심 비즈니스의 로직이 섞여서 유지보수가 어렵다.
- 시간을 측정하는 로직을 별도의 공통 로직으로 만들기 매우 어렵다.
- 시간을 측정하는 로직을 변경할 때 모든 로직을 찾아가면서 변경해야 한다.  
  예를 들어 밀리초씩 측정을 하고 있는데 나노초로 바꿔야 하면 일일이 변경해야 한다.

이러한 문제를 해결할 수 있는 것이 AOP라고 한다.

시간 측정하는 기능을 한 곳에 모아두고 관리하는 것이다.

---

## 6. 서블릿이란?

서블릿은 클라이언트의 요청을 처리하고, 그 결과를 반환하는 Servlet 클래스의 구현 규칙을 지킨 자바 웹 프로그래밍 기술이다.

자바를 사용하여 웹을 만들기 위해 필요한 기술이다.

클라이언트가 어떠한 요청을 하면 그에 대한 결과를 다시 전송해주어야 하는데, 이러한 역할을 하는 자바 프로그램이다.

---

### 서블릿 특징

- 클라이언트의 요청에 대해 동적으로 작동하는 웹 애플리케이션 컴포넌트이다.
- HTML을 사용하여 요청에 응답한다.
- Java Thread를 이용하여 동작한다.
- MVC 패턴에서 Controller로 이용된다.
- HTTP 프로토콜 서비스를 지원하는 `javax.servlet.http.HttpServlet` 클래스를 상속받는다.
- UDP보다 처리 속도가 느리다.
- HTML 변경 시 Servlet을 재컴파일해야 하는 단점이 있다.

---

### 서블릿 동작 흐름

1. 사용자, 즉 클라이언트가 URL을 입력하면 HTTP Request가 Servlet Container로 전송된다.
2. 요청을 전송받은 Servlet Container는 `HttpServletRequest`, `HttpServletResponse` 객체를 생성한다.
3. `web.xml`을 기반으로 사용자가 요청한 URL이 어느 서블릿에 대한 요청인지 찾는다.
4. 해당 서블릿에서 `service()` 메소드를 호출한 후 클라이언트의 GET, POST 여부에 따라 `doGet()` 또는 `doPost()`를 호출한다.
5. `doGet()` 또는 `doPost()` 메소드는 동적 페이지를 생성한 후 `HttpServletResponse` 객체에 응답을 보낸다.
6. 응답이 끝나면 `HttpServletRequest`, `HttpServletResponse` 두 객체를 소멸시킨다.

## 7. 미션: 스프링의 전반적인 요청, 응답 흐름을 설명하세요.

1. 클라이언트가 웹 애플리케이션 URL을 통해 HTTP 요청을 보낸다.

2. 이 요청은 가장 먼저 스프링 MVC 핵심 서블릿인 `DispatcherServlet`에 도착한다.

3. `DispatcherServlet`이 요청의 컨트롤러를 찾기 위해 `HandlerMapping`에게 문의한다.

4. `HandlerMapping`은 URL과 `@RequestMapping` 등이 매핑된 정보를 바탕으로 담당 컨트롤러를 찾아 반환한다.

5. `DispatcherServlet`은 공통된 방식으로 컨트롤러를 호출하기 위해 중간 다리 역할이 필요하다. 이를 위해 `HandlerAdapter`를 거쳐서 컨트롤러를 실행한다.

6. 어댑터는 컨트롤러가 어떤 형태, 즉 메서드 방식, 과거 서블릿 방식 등이든 상관없이 스프링이 일관되게 호출할 수 있도록 규격을 맞춰준다.

7. `HandlerAdapter`가 실제 Controller의 메서드를 호출한다.

8. 컨트롤러는 보통 개발자가 작성한 비즈니스 로직을 실행하여 데이터를 처리하고, 그 결과를 담아 응답할 대상, 즉 뷰 이름 또는 데이터 자체를 어댑터에 넘겨준다.

9. 컨트롤러가 작업을 마치면 결과 데이터인 Model과 화면 정보인 View Name을 담은 `ModelAndView` 객체를 `HandlerAdapter`를 통해 `DispatcherServlet`으로 돌려준다.

10. `DispatcherServlet`은 받은 뷰 이름을 실제 화면, 즉 HTML, JSP 등으로 렌더링하기 위해 `ViewResolver`를 호출한다.

11. `ViewResolver`는 문자열 이름에 앞뒤 경로인 Prefix, Suffix를 붙여 실제 물리적인 뷰 객체를 찾아낸다.

12. 선택된 View가 Model에 담긴 데이터를 활용해 최종 화면, 즉 HTML 등을 동적으로 그려내고 렌더링한다.

13. `DispatcherServlet`은 이 최종 결과를 클라이언트에게 HTTP 응답으로 전송하며 한 사이클이 마무리된다.
