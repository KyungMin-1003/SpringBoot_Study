# Chapter04

## 아키텍처 구조

### 정의
아키텍처 구조는 프로그램을 아무렇게나 만들지 않기 위해,
기능들을 역할별로 나누고 서로 연결하는 큰 설계 방식이다.

쉽게 말하면,

> 프로그램 전체를 어떻게 나누어 만들고,  
> 나눈 요소들이 어떻게 연결되는지 정한 구조이다.

---
### 아키텍처 구조에서 보는 것

아키텍처 구조는 다음과 같은 내용을 포함한다.

- 시스템의 구성 요소
- 구성 요소 간의 관계
- 시스템 외부 환경과의 관계
- 설계와 구현을 지원하는 구조
- 요구사항과 시스템 수명 주기
- 시스템 전체의 최적화

---

### 예시

코드를 짜기 전에 기능은 어디에 둘지, 서버랑 DB는 어떻게 연결할지 등등을 정하는 것이다. 

이런 아키텍처를 미리정하지 않고 한가지 파일에 다 넣을 수도 있지만 그렇게 된다면 문제가 생길 수 있다. 
기능이 많아지면 많아질 수록 오류가생긴다. 그래서 역할별로 나누는 것이다. 

웹서비스의 경우
- 사용자 화면
- 백엔드 서버
- 데이터베이스
- 파일 저장소
- 인증 모듈

이런 것들이 있다. 

화면 - 사용자에게 보여준다
서버: 요청을 처리함
DB: 데이터를 저장
인증 모튤: 로그인 확인

```text
사용자 → 화면 → 서버 → 데이터베이스
```


---


구조 분리를 잘한 아키텍처는 고치기 쉽고, 기능 추가 좋다.
계층형 아키텍처는 애플리케이션을 여러 층으로 나누어 관리한다.

예시

| 계층 | 역할 |
|---|---|
| Presentation 계층 | 사용자 요청을 받음 |
| Service 계층 | 주요 처리 흐름을 담당 |
| Repository 계층 | 데이터베이스 접근 담당 |
| Domain 계층 | 핵심 데이터와 비즈니스 규칙 담당 |


### 패키지 구조 예시

```text
com.example.application
├── controller    // Presentation 계층
├── service       // Service 계층
├── repository    // Repository 계층
└── model         // Domain 계층
```

### 장점

- 명확한 역할 분리: 각 계층이 명확한 역할을 가지므로, 코드를 이해하기 쉽다.
- 테스트 용이: 각 계층별로 독립적으로 테스트할 수 있어 편리합니다.
- 익숙한 패턴: 많은 개발자들이 이미 알고있는 패턴입니다.

### 단점

- 계층 간 의존성: 한 계층의 영향을 미칠 수 있다.
- 복잡성 증가: 큰 프로텍트에서는 계층의 수가 늘어나 복잡해질 수 있습니다.
- 비즈니스 로직의 분산: 비즈니스 로직이 여러 계층에 나뉘어 있을 수 있습니다.



## 도메인형 아키텍처란?

### 정의

도메인형 아키텍처는 시스템을 기술 중심이 아니라  
업무 기능 중심으로 나누는 구조이다.

쉽게 말하면,

> controller, service, repository처럼 기술별로 나누는 것이 아니라  
> user, order, product처럼 업무 기능별로 나누는 구조이다.


### 패키지 구조 예시

```html
com.example.application
├── user (사용자 도메인)
│   ├── UserController
│   ├── UserService
│   ├── UserRepository
│   └── User
├── order (주문 도메인)
│   ├── OrderController
│   ├── OrderService
│   ├── OrderRepository
│   └── Order
└── product (상품 도메인)
    ├── ProductController
    ├── ProductService
    ├── ProductRepository
    └── Product
```
---

### 장점

- 비즈니스 중심: 비즈니스 로직이 명확하게 드러나 이해하기 쉽다
- 유연성: 새로운 기능 추가 시 독립적으로 추가할 수 있다.
- 변경에 강함: 한 도메인의 변경이 다른 도메인에 영향을 미치지 않는다
---
### 단점

- 초기 학습 곡선: 익숙하지 않은 개발자에게 배우기 어려울 수 있다.

- 작은 프로젝트에 부적합: 작은 프로젝트에서는 불필요하게 복잡해 보일 수 있다.

- 도메인 간 협력 어려움: 여러 도메인이 협력해야 할 때 구조가 복잡해질 수 있다.

- 도메인형 아키텍처는 시스템을 기술 중심이 아니라 업무 기능 중심으로 나누는 구조이다.

- 아키텍처: 소프트웨어의 전체 구조와 구성 요소, 관계를 정한 설계 방식

- 도메인: 시스템이 해결하려는 실제 업무 영역

- 도메인형 아키텍처: 업무 기능 단위로 시스템을 나누는 구조


## DDD vs 도메인형 아키텍처

### DDD란?
DDD(Domain-Driven Design) 도메인 모델을 기반으로 복잡한 소프트웨어 시스템을 설계하고 개발하는 방법론이다.
쉽게 말하면,

> 시스템이 해결하려는 실제 업무를 깊게 이해하고,  
> 그 업무 규칙을 코드에 잘 반영하는 설계 방법이다.

**도메인(Domain)**

- 도메인은 소프트웨어 시스템이 해결하려는 문제 공간이다.
- 도메인은 비즈니스 로직과 규칙을 포함한다.
- 도메인은 여러 개의 서브도메인으로 구성될 수 있다.

**서브도메인(Subdomain)**

- 도메인의 하위 개념으로, 도메인을 더 작은 단위로 나눈 부분이다.
- 핵심 서브도메인(Core Subdomain): 비즈니스에서 가장 중요한 부분
- 지원 서브도메인(Supporting Subdomain): 핵심 서브도메인을 지원하는 부분
- 일반 서브도메인(Generic Subdomain): 다른 여러 도메인에서 일반적으로 사용되는 부분

**도메인 모델(Domain Model)**
- 도메인 전문가와 개발자가 공유하는 비즈니스 지식을 소프트웨어로 표현한 모델
- 엔테테, 밸류 오브젝트, 애그리게이드, 서비스 등으로 구성된다.

**엔티티(Entity)**
- 고유 식별자를 가진 객체
- 도메인 모델의 중요한 부분을 나타내며, 상태를 가지는 객체이다.

**밸류 오브젝트(Value Object)**
- 고유 식별자를 가진 객체
- 도메인 모델의 중요한 부분을 나타내며, 상태를 가지는 객체이다.
- 예: 돈, 날짜, 주소

**에그리게이트(Aggregate)**
- 관련된 엔티티와 밸류 오브젝트의 집합
- 트랜잭션의 경계를 정의한다. 
- 애그리게이트 루트(Aggregate Root): 애그리게이트의 진입점이 되는 엔티티로, 애그리게이트의 모든 변경은 애그리게이트 루트를 통해서만 이루어져야 한다.

**리포지토리(Repository)**
- 애그리게이트를 영구 저장소와 연결하는 인터페이스. 
- 도메인 객체를 저장하고 조회하는 책임을 가진다.

**도메인 서비스(Domain Service)**
- 애그리게이트를 영구 저장소와 연결하는 인터페이스.
- 도메인 객체를 저장하고 조회하는 책임을 가진다. 

**애플리케이션 서비스(Application Service)**
- 특정 엔티티에 속하지 않는 도메인 로직을 처리하는 서비스.
- 복잡한 비즈니스 로직을 캡슐화한다.

**도메인 이벤트(Domain Event)**
- 도메인 내에서 발생하는 사건.
- 다른 도메인에 알려야 할 중요한 변경 사항을 나타낸다.

**팩토리(Factory)**
- 복잡한 객체 생성 로직을 캡슐화하는 클래스.
- 엔티티나 애그리게이트를 생성하는 책임을 가진다.

| 구분 | DDD                    | 도메인형 아키텍처                       |
| -- | ---------------------- | ------------------------------- |
| 의미 | 도메인을 중심으로 설계하는 방법론     | 도메인 단위로 시스템을 나누는 구조             |
| 범위 | 더 넓음                   | 더 좁음                            |
| 초점 | 업무 개념, 규칙, 모델링         | 코드 구조, 폴더 구조, 모듈 분리             |
| 질문 | “이 업무의 핵심 규칙은 뭐지?”     | “이 기능 코드를 어디에 둘까?”              |
| 예시 | 주문 상태, 결제 규칙, 배송 조건 설계 | order, payment, delivery 폴더로 분리 |
| 핵심 | 도메인 모델                 | 도메인별 모듈 구조                      |
| 관계 | 철학/방법론                 | 구현 구조/아키텍처 스타일                  |

쉽게 정리

DDD는 가게 운영방식을 이해하는것에 가깝다.

주문은 어떻게 들어오는가

결제는 언제 완료되는가

재고는 언제 줄어드는가

이러한 업무 규칙을 이해하는것

도메인형 아키텍처는 
가게 부서를 나누는것에 가깝다

주문팀, 결제팀, 상품관리팀, 배송팀
역할별로 나눈 것이라고 생각하면 된다

둘은 서로 관련되어있지만 도메인형 아키텍처를 쓴다고 해서 무조건 DDD를 하는 건 아니다.

도메인별로 폴더를 나누는 것 = 도메인형 아키텍처
도메인의 개념과 규칙을 깊게 설계하는 것 = DDD
이렇게 이해하면 된다.

## Swagger
api 관련하면 open api specification(OSA)를 위한 프레임워크  OpenAPI에서 뺴놓을 수 없는 기능

API 에디터 혹은 API 매뉴얼 자동생성 및 테스트 사이트 생성

OSA는 RESTful 웹서비스를 약속된 규칙에 따라 약속된 규칙에 맞게 API 스펙을 json과 yaml 형식으로 표현한다.
이를 통해 직접 소스코드를 보거나 추가 문서 필요 없이 서비스를 이해할 수 있다.

쉽게말해 아래의 역할을 한다
API 주소가 뭐가 있는지 보여줌
GET, POST, PUT, DELETE 같은 요청 방식을 보여줌
어떤 값을 보내야 하는지 알려줌
응답 결과가 어떻게 생겼는지 보여줌
브라우저에서 직접 API를 실행해볼 수 있음

FastAPI에서 서버를 실행하고 /docs에 들어가면 나오는 화면이 바로 Swagger UI
실제로 소프트웨어 공학 과제할때 파이썬으로 fastAPI를 만들고 /docs에 들어가서 확인한 적이 있다.

여기서 OpenAPI와 Swagger의 차이를 알고 있어야한다.

OpenAPI는 API를 설명하는 표준 형식: 이 OpenAPI 파일에는 API 주소, 요청 방식, 파라미터, 요청 바디, 응답 형식, 인증 방식 같은 내용이 들어간다
Swagger: 그 OpenAPI 문서를 보기 좋게 보여주거나 편집하거나 코드로 바꾸는 도구 모음

**Swagger 화면에서 봐야 할 것들**
1. Endpoint - API 주소 ex) GET /user 여기서 /user이 endpoint이다.
2. Method - 요청 방식, GET,POST,PUT등등이 있다.
3. Parameters - API에 전달하는 값 예:GET /users/{user_id} 여기서 {user_id}는 path parameter이다.
4. Request Body -  POST나 PUT처럼 데이터를 보낼 때 사용하는 본문, 데이터들을 body에 보낸다. 
5. Response -API 실행 결과이다. 성공하면 성공했다고 응답을 한다. 

Swagger을 사용하면 API를 쉽게 테스트할 수 있다. 

Swagger은 FastAPI가 자동으로 OpenAPI문서를 만들고, Swagger UI가 그 문서를 읽어서 화면으로 보여주는 구조이다. 

FastAPI 코드 작성

↓

FastAPI가 OpenAPI 문서 자동 생성

↓

Swagger UI가 OpenAPI 문서를 읽음

↓

/docs 화면에 API 문서와 테스트 화면 표시

## 왜 DTO를 사용하는가

DTO는 **Data Transfer Object**의 약어로, 데이터 전송 객체를 의미한다.  
쉽게 말하면 **데이터 전달용 객체**이다.

---

### DTO가 필요한 이유

도메인 모델은 **Data Access 계층**에서 데이터가 저장되는 DB와 직접적으로 관련되어 있는 객체다.  
테이블의 구조에 맞게 설계되는 도메인 모델은 초기 설계 후 되도록 최대한 수정되지 않는 것이 안정적인 모델이다.

그에 반해 **Presentation 계층**에서는 View의 요구 사항에 따라 전달되는 데이터의 구조가 자주 바뀔 수 있다.

또한 도메인 모델은 비밀번호나 개인 정보 같이 View에서 노출하면 안 되는 민감한 정보를 가지고 있다.

이 객체들을 굳이 View에 전달할 필요도 없고, 전달해서는 안 되는 것들이다.

이는 각 계층의 관심사가 달라서 생기는 차이이다.

따라서 각 계층의 관심사에 맞게 데이터를 담는 객체도 구분을 지어줘야 한다.  
이를 **관심사의 분리**라고 한다.
---
### DTO의 핵심 역할

DTO의 핵심 관심사는 이름 그대로 데이터의 전달이다. DTO는 데이터를 담고, 다른 계층 또는 다른 컴포넌트들로
데이터들 넘겨주기 위한 자료구조이다.

---

### 1. 클라이언트와 서버의 호출을 최소화하기 위해서

DTO를 사용하는 이유중 하나는 클라이언트로부터 서버의 호출을 최소화하기 위해 

우리가 원격 인터페이스로 작업을 할 때, 호출에 따른 비용은 매우 비싸다. 그렇기 때문에 우리는 요청의 횟수들 줄여야 하고, 
이를 위해 한번의요청에 더 많은 데이터를 전송해야 한다. 

우리는 이를 수행하기 위해 많은 매개변수를 사용할 수 있지만 프로그래밍 하기 어렵고, 불가능 한 일이다. 

그렇기때문에 우리는 요청에 대한 모든 데이터를 보관할 수 있는 데이터 전송 객체, DTO를 만들어 사용한다. 

DTO주요 목적중 하나는 한 번의 호출로 여러 매개 변수를 일괄 처리해서 서버의 왕복을 줄이는 것이다.

### 2. 필요한 데이터만 주고받기 위해서

DB에 `User` 객체가 있다고 했을 때, `User`에는 다음과 같은 정보가 있을 수 있다.

```text
id
email
password
```

로그인한 사용자 정보를 응답할 때 비밀번호까지 보내면 안 된다.

그래서 응답용 DTO를 따로 만들어서 클라이언트에게 필요한 정보만 보낼 수 있게 한다.

즉, DTO를 쓰면 **보여주면 안 되는 데이터가 밖으로 나가는 것을 막을 수 있다.**

---

### 3. Entity를 직접 노출하지 않기 위해서

Entity는 보통 DB 테이블과 연결된 객체이다.

Entity를 직접 보내지 않고 DTO로 변환해서 보낸다.

```text
Entity → DTO → Client
```

---

### 4. 요청 데이터와 응답 데이터를 분리하기 위해서

회원가입할 때 받는 데이터와, 회원가입 후 응답하는 데이터는 다르다.

요청할 때는 `password`가 필요하지만, 응답할 때는 `password`를 보내면 안 된다.

그래서 요청용 DTO와 응답용 DTO를 따로 만든다.

```text
Request DTO
- email
- password
- name

Response DTO
- id
- email
- name
```

---

### 5. API 스펙을 명확히 만들기 위해서 

DTO를 사용하면 API가 어떤 데이터를 받는지, 어떤 데이터를 응답하는지 명확해진다.

DTO를 사용하면 필요한 정보를 보내줘서 어떤것이 필요한지 상대가 명확하게 알 수 없다. 
DTO가 요청/응답 데이터의 모양을 정해주는 역할을 한다.

DTO를 쓰면  Swagger만 보고 알 수 있다.

쉽게 정리: DTO없는 API는 회원가입할때 필요한 정보를 알아서 보내, DTO있는 API는 회원가입할때 필요한 정보를 알려주는 
느낌이다.

DTO를 사용하면 API가 어떤 데이터를 받고 어떤 데이터를 반환하는지 구조가 정해져서, 
프론트엔드와 백엔드가 헷갈리지 않고 같은 약속을 보고 개발할 수 있다.

## 6. 컨버터는 왜 사용하는가

### 정의
한 객체나 데이터 형식을 다른 객체나 데이터 형식으로 바꿔주는 역할
쉽게 데이터의 형태를 변환해주는거
---
이런 컨버터를 사용하는 이유는 각 계층에서 사용하는 객체의 모양이 다르기 때문에, 그 사이를 변환하기 위해 
사용한다.

컨버터가 없으면 Controller나 Service안에 변환코드가 섞이게 된다.

```text
Controller가 너무 복잡해짐
Service도 변환 코드 때문에 지저분해짐
같은 변환 코드를 여러 번 반복하게 됨
응답 형식이 바뀌면 여러 곳을 수정해야 함
```
컨버터를 쓰면 컨버터 클래스를 따로 만들어서 변환코드를 한곳에 모을 수 있다.

---

### 핵심역할
```text
Request DTO → Entity
Entity → Response DTO
Domain Model → DTO
DTO → Domain Model
```

예를 들어 회원가입이면:

SignupRequestDto → UserEntity

조회 응답이면:

UserEntity → UserResponseDto

이렇게 바꿔준다.

3. 수정이 쉬워지기 위해
4. Entity 직접 노출을 막기 위해
