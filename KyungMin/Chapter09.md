# Chapter 09. 인증과 인가 개념 정리

## 1. 세션과 토큰의 차이는?

세션과 토큰은 로그인한 사용자의 인증 상태를 유지하는 대표적인 방법이다. 가장 큰 차이는 인증 정보를 저장하고 관리하는 위치에 있다.

### 세션 기반 인증

세션 방식에서는 사용자의 로그인 상태를 서버가 관리한다.

1. 사용자가 아이디와 비밀번호로 로그인한다.
2. 서버가 사용자를 확인하고 세션을 생성한다.
3. 서버는 생성된 세션을 세션 저장소에 보관한다.
4. 클라이언트는 `Session ID`를 쿠키로 전달받는다.
5. 이후 요청마다 쿠키에 담긴 `Session ID`를 서버에 전달한다.
6. 서버는 세션 저장소에서 해당 세션을 조회하여 사용자를 인증한다.

```text
클라이언트 ── Session ID ──> 서버 ── 세션 조회 ──> 세션 저장소
```

#### 장점

- 인증 정보를 서버가 관리하므로 제어하기 쉽다.
- 서버에서 세션을 삭제하면 즉시 로그아웃시킬 수 있다.
- 클라이언트에 사용자 정보가 직접 저장되지 않는다.

#### 단점

- 사용자가 많아질수록 서버의 세션 저장 공간이 필요하다.
- 여러 서버를 사용하는 환경에서는 세션을 공유하기 위한 별도의 저장소가 필요하다.
- 서버 확장 시 세션 동기화와 관리가 복잡해질 수 있다.

---

### 토큰 기반 인증

토큰 방식에서는 로그인 성공 시 서버가 인증 정보를 담은 토큰을 발급한다. 클라이언트는 토큰을 보관하고, 이후 요청마다 서버에 전달한다.

1. 사용자가 아이디와 비밀번호로 로그인한다.
2. 서버가 사용자를 확인한 뒤 토큰을 발급한다.
3. 클라이언트가 발급받은 토큰을 보관한다.
4. 이후 요청마다 HTTP 헤더에 토큰을 담아 전달한다.
5. 서버가 토큰의 유효성을 검사하여 사용자를 인증한다.

일반적으로 다음과 같은 형태로 전달한다.

```http
Authorization: Bearer {Access-Token}
```

JWT를 사용하면 토큰 내부에 사용자 식별자, 권한, 만료 시간 등의 정보를 포함할 수 있다.

```text
클라이언트 ── JWT ──> 서버 ── 서명 및 만료 시간 검증 ──> 인증 완료
```

#### 장점

- 서버가 모든 사용자의 인증 상태를 직접 보관하지 않아도 된다.
- 여러 서버에서 동일한 방식으로 토큰을 검증할 수 있어 확장에 유리하다.
- 웹, 모바일 앱 및 다른 서비스에서도 사용할 수 있다.
- 마이크로서비스 구조에 적용하기 편리하다.

#### 단점

- 발급된 토큰을 만료 전에 즉시 무효화하기 어렵다.
- 토큰이 탈취되면 공격자가 사용자처럼 요청할 수 있다.
- JWT에 정보가 많아지면 요청 데이터의 크기가 커진다.
- 토큰의 저장 위치와 갱신 정책을 안전하게 설계해야 한다.

### 세션과 토큰 비교

| 구분 | 세션 방식 | 토큰 방식 |
|---|---|---|
| 인증 상태 관리 | 서버 | 주로 클라이언트가 토큰 보관 |
| 요청 시 전달값 | Session ID | Access Token |
| 서버 저장소 | 일반적으로 필요 | Access Token만 사용할 경우 필수는 아님 |
| 서버 확장 | 세션 공유 문제 발생 가능 | 상대적으로 유리 |
| 강제 로그아웃 | 세션 삭제로 쉽게 처리 | 별도의 차단 정책이 필요 |
| 탈취 위험 | Session ID 탈취에 주의 | 토큰 탈취에 주의 |
| 대표적인 사용 | 전통적인 웹 서비스 | REST API, 모바일 앱, MSA |

> 세션은 쿠키와 반대되는 개념이 아니다. 쿠키는 데이터를 전달하거나 저장하는 수단이며, 세션 방식에서는 일반적으로 쿠키에 `Session ID`를 저장한다. 토큰 역시 설계에 따라 쿠키에 저장할 수 있다.

## 2. 액세스 토큰과 리프레시 토큰이란?

토큰 기반 인증에서는 보안성과 사용자 편의성을 모두 확보하기 위해 액세스 토큰과 리프레시 토큰을 함께 사용할 수 있다.

### 액세스 토큰(Access Token)

액세스 토큰은 사용자가 보호된 API에 접근할 때 사용하는 토큰이다.

예를 들어 마이페이지 조회 요청을 보낼 때 다음과 같이 전달한다.

```http
GET /api/members/me
Authorization: Bearer {Access-Token}
```

서버는 액세스 토큰을 검사하여 다음 내용을 확인한다.

- 서버가 발급한 정상적인 토큰인가?
- 토큰이 변조되지 않았는가?
- 토큰의 유효 기간이 지나지 않았는가?
- 해당 요청을 수행할 권한이 있는가?

액세스 토큰은 요청마다 사용되므로 탈취 위험을 줄이기 위해 유효 기간을 비교적 짧게 설정한다.

### 리프레시 토큰(Refresh Token)

리프레시 토큰은 만료된 액세스 토큰을 새로 발급받기 위해 사용하는 토큰이다.

액세스 토큰이 만료될 때마다 사용자가 아이디와 비밀번호를 다시 입력하면 불편하다. 따라서 로그인할 때 액세스 토큰과 함께 리프레시 토큰을 발급하고, 액세스 토큰이 만료되면 리프레시 토큰을 이용하여 새로운 액세스 토큰을 요청한다.

```text
1. 로그인 성공
   → Access Token + Refresh Token 발급

2. Access Token으로 API 요청
   → 마이페이지 등 보호된 기능 이용

3. Access Token 만료
   → API 요청 실패

4. Refresh Token을 갱신 API에 전달
   → Refresh Token 검증

5. 새로운 Access Token 발급
   → 로그인 상태 유지
```

리프레시 토큰은 액세스 토큰보다 일반적으로 유효 기간이 길다. 따라서 더욱 안전하게 보관해야 하며, 서버의 데이터베이스나 Redis 등에 저장하여 로그아웃 및 탈취 상황에 대응할 수 있다.

### 액세스 토큰과 리프레시 토큰 비교

| 구분 | 액세스 토큰 | 리프레시 토큰 |
|---|---|---|
| 목적 | 보호된 API 사용 | 액세스 토큰 재발급 |
| 사용 빈도 | API 요청마다 사용 | 액세스 토큰 갱신 시 사용 |
| 유효 기간 | 비교적 짧음 | 비교적 김 |
| 서버 저장 | 반드시 필요한 것은 아님 | 보안상 저장하는 경우가 많음 |
| 탈취 시 위험 | 만료 전까지 API 접근 가능 | 새로운 액세스 토큰을 계속 발급받을 수 있음 |
| 전달 범위 | 여러 보호 API | 토큰 갱신 API에서만 사용 |

### 두 토큰을 분리하는 이유

액세스 토큰 하나의 유효 기간을 길게 설정하면 사용자 편의성은 좋아지지만, 토큰이 탈취되었을 때 오랫동안 악용될 수 있다.

반대로 액세스 토큰의 유효 기간을 너무 짧게 설정하면 안전성은 높아지지만 사용자가 자주 로그인해야 한다. 따라서 다음과 같이 역할을 분리한다.

- 액세스 토큰: 짧게 사용하여 탈취 피해를 줄인다.
- 리프레시 토큰: 액세스 토큰을 재발급하여 로그인 상태를 유지한다.

### 주의할 점

- JWT의 Payload는 암호화된 영역이 아니라 Base64 방식으로 인코딩된 영역이다.
- 따라서 비밀번호, 주민등록번호 등 민감한 정보를 JWT에 넣으면 안 된다.
- HTTPS를 사용하여 전송 중 토큰 탈취를 방지해야 한다.
- 리프레시 토큰은 액세스 토큰보다 더 엄격하게 관리해야 한다.
- 로그아웃 시 서버에 저장된 리프레시 토큰을 삭제하거나 폐기 처리해야 한다.
- 재발급 시 기존 리프레시 토큰을 새 토큰으로 교체하는 `Refresh Token Rotation`을 적용할 수 있다.

## 3. OAuth 1.0과 OAuth 2.0의 차이는?

OAuth는 사용자의 비밀번호를 외부 서비스에 직접 제공하지 않고도 특정 자원에 접근할 권한을 위임하기 위한 표준이다.

예를 들어 사용자가 우리 서비스에서 Google 로그인을 선택하면, 우리 서비스가 사용자의 Google 비밀번호를 직접 전달받는 것이 아니다. Google이 사용자를 인증한 뒤 우리 서비스에 제한된 접근 권한을 부여한다.

> OAuth의 핵심 목적은 인증 자체보다 권한 위임이다. 소셜 로그인처럼 사용자 인증에 활용하려면 OAuth 2.0과 함께 사용자 신원 정보를 다루는 OpenID Connect(OIDC)를 사용할 수 있다.

### OAuth 1.0

OAuth 1.0은 각 API 요청에 서명을 추가하는 방식이다.

클라이언트는 요청 정보와 비밀키 등을 이용하여 서명을 만들고, 서버는 해당 서명을 다시 계산하여 요청이 변조되지 않았는지 확인한다.

#### 특징

- 요청마다 암호화 서명을 생성한다.
- 요청의 위조 및 변조를 검증할 수 있다.
- 토큰이 노출되더라도 서명 생성에 필요한 정보가 없으면 사용하기 어렵다.
- 서명 생성 및 검증 과정이 복잡하다.
- 웹뿐 아니라 모바일 등 다양한 클라이언트 환경에 적용하기 불편하다.

### OAuth 2.0

OAuth 2.0은 OAuth 1.0의 복잡한 서명 방식을 단순화하고 다양한 애플리케이션 환경을 지원하도록 설계되었다.

일반적으로 액세스 토큰을 `Bearer Token` 형태로 전달한다.

```http
Authorization: Bearer {Access-Token}
```

Bearer Token은 토큰을 가지고 있는 사람이 권한을 사용할 수 있다는 의미다. 따라서 OAuth 2.0에서는 토큰 자체의 요청별 서명보다는 HTTPS를 통한 안전한 통신이 매우 중요하다.

#### 특징

- 요청마다 복잡한 서명을 생성하지 않아도 된다.
- HTTPS를 기반으로 토큰을 안전하게 전달한다.
- 액세스 토큰과 리프레시 토큰을 사용할 수 있다.
- 웹, 모바일 앱 및 서버 간 통신 등 다양한 환경을 지원한다.
- 권한 범위를 `Scope`로 제한할 수 있다.
- 현재 대부분의 소셜 로그인과 API 권한 위임에서 사용된다.

### OAuth 1.0과 OAuth 2.0 비교

| 구분 | OAuth 1.0 | OAuth 2.0 |
|---|---|---|
| 요청 보호 방식 | 요청마다 서명 생성 | 주로 Bearer Token과 HTTPS 사용 |
| 구현 난이도 | 높음 | 상대적으로 낮음 |
| 토큰 종류 | 주로 요청 토큰과 액세스 토큰 | 액세스 토큰, 선택적으로 리프레시 토큰 |
| 유효 기간 | 표준화가 제한적 | 만료 및 갱신 구조 지원 |
| 권한 범위 | 비교적 단순 | Scope로 세분화 가능 |
| 클라이언트 지원 | 적용이 복잡함 | 웹, 모바일, SPA 등 다양한 환경 지원 |
| 확장성 | 낮은 편 | 높은 편 |
| 현재 활용도 | 거의 사용되지 않음 | 널리 사용됨 |

### OAuth 2.0의 주요 참여자

- `Resource Owner`: 자원의 소유자인 사용자
- `Client`: 사용자의 권한을 위임받으려는 애플리케이션
- `Authorization Server`: 사용자를 인증하고 토큰을 발급하는 서버
- `Resource Server`: 토큰을 확인하고 보호된 자원을 제공하는 API 서버

### OAuth 2.0의 기본 흐름

```text
사용자
  ↓ 소셜 로그인 요청
클라이언트 애플리케이션
  ↓ 인증 페이지로 이동
Authorization Server
  ↓ 사용자 로그인 및 권한 동의
클라이언트 애플리케이션
  ↓ 인가 코드를 토큰으로 교환
Authorization Server
  ↓ Access Token 발급
클라이언트 애플리케이션
  ↓ Access Token으로 사용자 정보 요청
Resource Server
```

최근 웹 및 모바일 서비스에서는 주로 `Authorization Code` 방식이 사용된다. 공개 클라이언트나 보안을 강화해야 하는 환경에서는 인가 코드를 가로채는 공격을 막기 위해 `PKCE`를 함께 적용한다.

## 핵심 요약

- 세션 방식은 로그인 상태를 서버가 저장하고, 클라이언트는 주로 Session ID를 전달한다.
- 토큰 방식은 클라이언트가 토큰을 보관하고 요청마다 서버에 전달한다.
- 액세스 토큰은 실제 API를 사용할 때 제출하는 수명이 짧은 토큰이다.
- 리프레시 토큰은 액세스 토큰을 다시 발급받기 위한 수명이 긴 토큰이다.
- JWT는 토큰의 한 가지 형식이며, 토큰 기반 인증이 반드시 JWT를 의미하는 것은 아니다.
- OAuth는 비밀번호를 공유하지 않고 다른 서비스에 접근 권한을 위임하기 위한 표준이다.
- OAuth 1.0은 요청마다 서명을 생성하여 안전하지만 구현이 복잡하다.
- OAuth 2.0은 Bearer Token과 HTTPS를 중심으로 동작하며 다양한 클라이언트 환경을 지원한다.
- OAuth 2.0만으로 로그인을 설명하기보다는 사용자 인증 계층인 OpenID Connect와 함께 이해하는 것이 정확하다.
