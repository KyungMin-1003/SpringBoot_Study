# 스프링 입문 강의 요약

## 1. 스프링 웹 개발 기초

스프링 웹 개발 방식은 크게 세 가지로 나눌 수 있다.

### 1) 정적 컨텐츠

정적 컨텐츠는 서버에서 파일을 그대로 보내주는 방식이다.

예를 들어, 컨테이너에 관련된 컨트롤러가 없다면 서버는 HTML 파일을 그대로 찾아서 브라우저에 전달한다.

### 2) MVC와 템플릿 엔진

MVC와 템플릿 엔진 방식은 JSP, PHP처럼 HTML을 서버에서 프로그래밍하여 동적으로 바꿔서 제공하는 방식이다.

MVC는 다음 세 가지로 구성된다.

- Model
- View
- Controller

과거에는 이 세 가지 역할을 한 곳에 합쳐서 사용했지만, 그렇게 하면 효율성이 떨어지고 유지보수가 어려웠다.  
그래서 역할을 나누어 관리하게 되었고, 이를 통해 효율성과 유지보수성이 좋아졌다.

MVC와 템플릿 엔진의 동작 과정은 다음과 같다.

1. 브라우저에서 요청을 보낸다.
2. 요청이 컨트롤러로 전달된다.
3. 컨트롤러가 viewResolver로 전달한다.
4. viewResolver가 템플릿을 찾는다.
5. Thymeleaf 템플릿 엔진이 처리한다.
6. 렌더링된 HTML을 브라우저에 반환한다.

### 3) API

API는 주로 JSON을 사용해서 사용자와 서버가 데이터를 주고받는 방식이다.

문자열 데이터를 반환하면 `StringConverter`가 동작한다.  
이 경우 별다른 가공 없이 문자열이 그대로 반환된다.

하지만 데이터가 객체라면 `JacksonConverter`가 동작하여 객체를 JSON 형식으로 변환한다.

## 2. 회원 관리 예제 - 백엔드 개발

회원 관리 예제에서 백엔드는 다음과 같은 구조로 나뉜다.

### 1) 컨트롤러

컨트롤러는 웹 MVC에서 컨트롤러 역할을 담당한다.

사용자의 요청을 받고, 필요한 처리를 서비스에 전달한 뒤 결과를 반환한다.

### 2) 서비스

서비스는 핵심 비즈니스 로직을 구현하는 부분이다.

예를 들어 회원 가입 시 중복 회원을 제한하는 로직 같은 것이 서비스에 들어간다.

### 3) 리포지토리

리포지토리는 데이터베이스에 접근하는 역할을 한다.

도메인 객체를 데이터베이스에 저장하고 관리한다.

### 4) 도메인

도메인은 비즈니스 도메인 객체를 의미한다.

예를 들어 회원 관리 예제에서는 `Member` 같은 객체가 도메인에 해당한다.

## 3. 테스트 케이스 작성

코드가 정상적으로 작동하는지 확인하기 위해 테스트 케이스를 작성할 필요가 있다.

테스트 방식 중 하나로 TDD 방식이 많이 사용된다.

### TDD란?

TDD는 `Test-Driven Development`의 약자로, 테스트 주도 개발을 의미한다.

TDD는 다음 세 단계로 진행된다.

### 1) Red

구현하려는 기능에 대한 테스트 코드를 먼저 작성한다.  
아직 실제 코드가 없거나 부족하기 때문에 테스트는 실패한다.

### 2) Green

테스트를 통과할 수 있을 만큼의 최소한의 실제 코드를 작성한다.  
이 단계에서는 테스트 성공을 목표로 한다.

### 3) Refactor

테스트가 성공했다면 코드를 깔끔하게 정리한다.  
기능은 유지하면서 코드의 구조나 가독성을 개선한다.

---

## 4. 테스트 코드 작성

테스트 코드를 작성할 때는 `@Test` 어노테이션을 사용한다.

테스트를 하나 끝내면 데이터를 초기화해줄 필요가 있다.  
이를 위해 `@AfterEach`를 사용할 수 있다.

### @AfterEach

`@AfterEach`는 테스트 메서드 하나가 끝날 때마다 실행되는 기능이다.

따라서 테스트가 끝난 뒤 데이터를 지워주는 코드를 작성할 수 있다.

### @Transactional

하지만 `@Transactional`을 이용하면 더 편하게 테스트 데이터를 관리할 수 있다.

`@Transactional`은 각 테스트 케이스가 끝날 때마다 데이터를 자동으로 롤백해주는 기능이다.  
따라서 수동으로 `clear` 작업을 할 필요가 없다.

테스트는 상황에 따라 다르지만, 작게 쪼개서 하나하나 실행해보는 것이 좋다고 한다.

## 5. 스프링 빈과 의존관계

스프링 빈은 스프링 컨테이너에 의해 관리되는 자바 객체를 의미한다.

스프링 빈을 등록하는 방법은 크게 두 가지가 있다.

1. 컴포넌트 스캔과 자동 의존관계 설정
2. 자바 코드로 직접 스프링 빈 등록

두 가지 방법 모두 알아두어야 한다.

---

## 6. 컴포넌트 스캔과 자동 의존관계 설정

컴포넌트 스캔을 사용하는 방법은 스프링에서 `@Component` 어노테이션을 사용하는 것이다.

`@Component` 어노테이션이 붙어 있으면 스프링이 자동으로 해당 클래스를 스프링 빈으로 등록한다.

다음 어노테이션들도 따로 등록할 필요가 없다.

- `@Controller`
- `@Service`
- `@Repository`

그 이유는 이 세 가지 어노테이션 안에 `@Component`가 포함되어 있기 때문이다.

---

## 7. @Autowired와 의존성 주입

`@Autowired`를 이용하면 의존성을 주입할 수 있다.

생성자나 필드에 `@Autowired`를 붙이면, 스프링이 컨테이너에 등록된 빈들 중에서 필요한 객체를 자동으로 찾아서 넣어준다.

의존성 주입에는 세 가지 방법이 있다.

1. 생성자 주입
2. 필드 주입
3. Setter 주입

이 중에서는 생성자 주입을 권장한다.

---

## 8. 자바 코드로 직접 스프링 빈 등록하기

두 번째 방법은 새로운 클래스를 만들어서 직접 스프링 빈을 등록하는 방법이다.

이때 `@Bean` 어노테이션을 사용해서 직접 등록한다.

## 9. AOP

AOP는 공통 관심 사항과 핵심 관심 사항을 분리하기 위해 사용된다.

### AOP가 필요한 상황

예를 들어 모든 메소드의 실행 시간을 측정하고 싶을 때 AOP가 필요할 수 있다.

회원 가입 시간이나 회원 조회 시간을 측정하고 싶다고 가정해보자.

이때 시간 측정 기능은 핵심 비즈니스 로직이 아니다.  
회원 가입이나 회원 조회 자체가 핵심 로직이고, 시간 측정은 공통 관심 사항이다.

### AOP가 필요한 이유

이런 경우 다음과 같은 문제가 생긴다.

- 회원 가입, 회원 조회에 시간을 측정하는 기능은 핵심 관심 사항이 아니다.
- 시간을 측정하는 로직은 공통 관심 사항이다.
- 시간을 측정하는 로직과 핵심 비즈니스 로직이 섞여서 유지보수가 어렵다.
- 시간을 측정하는 로직을 별도의 공통 로직으로 만들기 어렵다.
- 시간을 측정하는 로직을 변경할 때 모든 로직을 찾아가면서 변경해야 한다.
    - 예를 들어 밀리초 단위로 측정하고 있는데 나노초 단위로 바꿔야 한다면 일일이 변경해야 한다.

이러한 문제를 해결하기 위해 AOP가 생겼다.

AOP는 공통 관심 사항과 핵심 관심 사항을 분리하는 것이다.  
예를 들어 시간 측정 로직 같은 경우, 이 로직을 한 곳에 모아두고 원하는 곳에 지정해서 적용할 수 있다.

### AOP의 동작 방법

스프링은 실제 서비스 대신 가짜 프록시를 만들어 앞에 세운다.

프록시에서 공통 로직을 먼저 처리한 뒤 실제 서비스에 연결되는 방식으로 동작한다.

# 10. SQL 기초 - DDL vs DML

## 관계형 데이터베이스

관계형 데이터베이스는 RDBMS라고도 한다.

RDBMS는 `Relational Database Management System`의 약자이다.

관계형 데이터베이스는 데이터를 테이블로 구성하고, 이 테이블들 간의 관계를 정의하여 데이터를 저장하는 방식을 의미한다.

관계형 데이터베이스는 다음과 같은 구조로 이루어져 있다.

- 테이블
- 행
- 열

데이터를 쉽게 저장하고 조회할 수 있다는 장점이 있다.

데이터베이스 관리 시스템의 주요 컴포넌트로는 다음과 같은 것들이 있다.

- DDL
- DML
- DCL
- TCL

---

## 11. DDL 데이터 정의 언어

DDL은 `Data Definition Language`의 약자이다.

DDL은 데이터베이스에서 스키마를 정의하거나 수정하고, 테이블, 뷰, 인덱스 등의 데이터 구조를 생성하거나 변경하는 데 사용된다.

일반적으로 데이터베이스 관리 시스템(DBMS)에서 제공하는 SQL 명령어를 사용하여 DDL 작업을 수행한다.

또한 DDL은 데이터베이스의 구조를 정의하기 때문에 데이터베이스 설계 및 관리에 매우 중요한 역할을 한다.

### DDL 명령어

| 명령어 | 설명 |
|---|---|
| CREATE | 데이터베이스 객체 생성 시 사용 |
| ALTER | 데이터베이스 객체를 수정하는 데 사용 |
| DROP | 데이터베이스 객체 삭제, 신중한 사용 필요 |
| RENAME | 이름 변경 |
| TRUNCATE | 테이블의 모든 데이터를 삭제하는 데 사용, 모든 행이 삭제되지만 테이블 자체는 사라지지 않음 |
| COMMENT | 객체에 대한 주석을 추가하는 데 사용 |

### DDL 코드 예시

```sql
-- 새로운 데이터베이스 생성
CREATE DATABASE Hello_db;

-- 테이블 생성
CREATE TABLE member (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE
);

-- 테이블 삭제
DROP TABLE member;
