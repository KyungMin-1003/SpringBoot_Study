## 테스트케이스 작성

코드가 정상적으로 작동하는지 확인하기 위해서 테스트케이스를 작성할 필요가 있다.

TDD방식을 많이 사용하는데 TDD(Test-Driven Development) '테스트 주도 개발'이라고 한다.
Red - 구현하려는 기능에 대한 테스트 코드를 먼저 짠다.
Green - 테스트를 통과할 수 있을 만큼의 최소한의 실제 코드를 작성한다. 성공을 목표로한다.
Refactor - 테스트가 성공했다면 코드를 깔끔하게 정리한다.

##테스트코드 작성
@Test 어노테이션을 사용
테스트를 하나 끝내면 데이터를 초기화해줄 필요가 있다. @AfterEach사용해서 메서드 하나 끝나면 데이터를 지워주는 코드를 작성
하지만 @Transactional이용하면 더 편하다.

@Transactional-각 테스트케이스가 끝날 때마다 데이터를 자동으로 롤백해주는 기능. 수동으로 clear작업을 할 필요가 없다.
@AfterEach - 메서드 실행이 끝나면 실행되는 것

이러한 테스트는 사오항에 따라 다르지만 작개 쪼개서 하나하나 실행해보는것이 좋다고 한다.

### 스프링 빈과 의존관

스프링 빈은 컨테이너에 의해 관리되는 자바 객체를 의미한다.
스프링 빈을 등록하는 2가지 방법이 존재한다.

첫번째는 컴포넌트 스캔과 자동 의존관계 설정이다.
두번쨰는 자바 코드로 직접 스프링 빈 등록하기가 있다.
두 가지 방법을 모두 알아두어야 한다.

컴포넌트 스캔을 쓰는 방법은 스프링에 @Component 어노테이션을 사용하는 방법이다. @Component 어노테이션이 존재하면
spring에서 자동으로 스프링 빈으로 등록된다

@Controller
@Service
@Repository
이 세가지는 따로 등록이 필요 없는데 그 이유는 이 세가지 어노테이션 안에 @Component가 들어가 있기 때문이다.

@Autowired를 이용해 의존성을 주입한다. 생성자나 필드에 @Autowired를 붙이면, 스프링이 컨테이너에 등록된 빈들중에서 필요한 객체를 자동으로 찾아서넣어준다

두번째 방법은 새로운 클래스를 만들어서 직접 주입하는 방법이다. 이때 @Bean어노테이션을 써서 직접 등록한다.

의존성 주입에는 세가지 방법이 있는데
생성자 주입
필드 주입
setter주입이 있는데

생성자 주입을 권장한다.

### AOP

AOP가 필요한 상황
모든 메소드의 시간을 측정하고 싶을 때
공통관심사항 vs 핵심관심사항
회원 가입 시간, 회원 조회 시간을 측정하고 싶음

이런경우 문제들

- 회원가입, 회원 조회에 시간을 측정하는 기능은 핵심 관심 사항이 아니다
- 시간을 측정하는 로직은 공통 관심 사항이다.
- 시간을 측정하는 로직과 핵심 비즈니스의 로직이 섞여서 유지보수가 어렵다.
- 시간을 측정하는 로직을 별도의 공통 로직으로 만들기 매우 어렵다.
- 시간을 측정하는 로직을 변경할 때 모든 로직을 찾아가면서 변경해야 한다 → 밀리초씩 측정을 하고 있는데 나노초로 바꿔야하면 일일이 변경해야함.

이러한 문제들이 있어서 AOP가 생겼다.
공통관심사항과 핵심관심사항을 분리하는것이다. 예시로 시간측정로직같은 경우 이 로직을 한곳에 몰아넣고
원하는 곳에 지정해서 적용할 수 있는 방법이다.

AOP의 동작방법 - spting는 실제 service대신 가자 프록시를 만들어서 앞에 세운다. 프록시에서 처리를 한 뒤에 실제 서비스에 연결되는 방식이다.

# 2. SQL 기초 DDL vs DML

관계형 데이터베이스(RDBMS: Relational Database Management System):데이터를 테입블로 구성하고 이 테이블들 간의 관계를 정의하여 데이터를 저장하는 방식을 의미함.

- 테이블, 행 열로 이루어져 있으며 데이터를 쉽게 저장하고 조회할 수 있단 장점을 가지고 있음
- 데이터베이스 관리 시스템의 주요 컴포넌트로는 DDL DML, DCL, TCl이 있음.

## DDL 데이터 정의 언어 (DDL: Data Definition Language)

- 데이터베이스에서 스키마를 정의하거나 수정하고, 테이블, 뷰, 인덱스 등의 데이터 구조를 생성하거나 변경하는 데 사용됩니다.
- 일반적으로 데이터베이스 관리 시스템(DBMS)에서 제공하는 SQL 명령어를 사용하여 DDL 작업을 수행합니다.
또한 데이터베이스의 구조를 정의하기 때문에 데이터베이스의 설계 및 관리에 매우 중요한 역할을 합니다.

CREATE - 데이터베이스 객체 생성시 사용
ALTER - 데이터 베이스 객체를 수정하는데 사용
DROP - 데이터베이스 객체 삭제 (신중한 사용필요)
RENAME -이름 변경
TURNCATE -테이블의 모든 데이터를 삭제하는데 사용 (모든행이 삭제 테이블자체는 사라지지않는다)
COMMENT - 객체에 대한 주석을 추가하는데 사용

코드
-- 새로운 데이터베이스 생성
CREATE DATABASE Hello_db;

- -테이블 생성
CREATE TABLE member (
id BIGINT AUTO_INCREMENT PRIMARY KEY,
name VARCHAR(255) NOT NULL,
email VARCHAR(255) UNIQUE
);
--테이블 삭제
DROP TABLE member;

##DML 데이터 조작 언어(DML: Data Manipulation Language)
-데이터베이스에서 저장된 데이터를 조회, 삽입, 수정, 삭제하는 작업에 사용됩니다. 
DML은 주로 데이터베이스 관리 시스템(DBMS)에서 제공하는 SQL 명령어를 사용하여 수행됩니다.

SELECT - 데이터 베이스에서 데이터를 조회할 때 사용하는 명령어로 특정 테이블의 열을 선택하여 결과를 반환
INSERT - 데이터베이스의 테이블에 새로운 행을 삽입하는 명령어로 지정된 열에 값을 추가합니다.
UPDATE - 데이터베이스의 테이블에서 특정 행의 값을 수정하는 명령어로, 지정된 열의 값을 업데이트합니다.
DELETE - 데이터베이스의 테이블에서 특정 행을 삭제하는 명령어로 지정된 조건에 맞는 행을 제거합니다
MERGE - 두 개의 테이블을 조합하여 새로운 테이블을 만들거나, 기존의 테이블의 업데이트,삽입,삭제를 수행하는 명령어
CALL - 저장 프로시저나 함수를 호출하는 명령어로, 특정한 작업을 수행하기 위해 저장 프로시저나 함수를 실행합니다.
EAPLAIN PLAN - SQL 문의 실행 계획을 분석하여 최적화할 수 있는 정보를 제공하는 명령어
LOCK TABLE - 테이블에 대한 잠금을 설정하여 다른 사용자가 해당 테이블을 수정하지 못하도록하는 명령어입니다.

- 추가
update 는 이미 존재하는 데이터만 대상으로, 대상이 확실히존재할 때주로 사용
merge는 조건에따라 수정할지, 새로넣을지 결정,데이터 동기화 외부시스템에서 받은 데이터를 우리 db에 반영할 때,
일괄처리 수만 건의 데이터를 한꺼번에 대조해서 변경 사항만 업데이트하고 싶을 때 사용

EAPLAIN PLAN - 옵티마이저가 경로계산한걸 사용자에게 공개함.
LOCK TABLE - 사용이유는 동시에 여러 트랜잭션이 동일한 테이블에서 작업을 수행하되 서로 충돌하지 않도록 제어하고 데이터 무결성을 보장하는 것임.

##DDL DML 차이
DDL은 데이터베이스 구조를 변경하는데사용된다. DML은 데이터베이스 안에있는 데이터 조회나 삽입등 안의 데이터를 다룬다.
서로의 역할과 사용하는 범위의 차이가 있다.
