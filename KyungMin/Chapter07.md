# Page와 Slice

## 1. 정의

`Page`와 `Slice`는 **Spring Data JPA에서 페이징 처리를 할 때 사용하는 반환 타입**이다.

페이징이란 데이터를 한 번에 전부 가져오지 않고, 일정 개수만 잘라서 가져오는 방식이다.

예를 들어 게시글이 1,000개 있을 때 한 번에 전부 조회하는 것이 아니라, 10개씩 나누어 조회하는 방식이다.

---

## 2. Page

`Page`는 현재 페이지의 데이터뿐만 아니라 **전체 데이터 개수, 전체 페이지 수** 같은 정보까지 함께 제공한다.

```java
Page<Post> posts = postRepository.findAll(pageable);
```

### 자주 사용하는 메서드

| 메서드                  | 의미             |
| -------------------- | -------------- |
| `getContent()`       | 현재 페이지의 실제 데이터 |
| `getTotalElements()` | 전체 데이터 개수      |
| `getTotalPages()`    | 전체 페이지 수       |
| `getNumber()`        | 현재 페이지 번호      |
| `getSize()`          | 페이지 크기         |
| `hasNext()`          | 다음 페이지 존재 여부   |
| `hasPrevious()`      | 이전 페이지 존재 여부   |

### Page의 특징

`Page`는 전체 데이터 개수를 알아야 하기 때문에 내부적으로 **count 쿼리**가 추가로 실행된다.

즉, 실제 데이터를 조회하는 쿼리와 전체 개수를 조회하는 쿼리가 함께 실행될 수 있다.

따라서 전체 페이지 수가 필요한 화면에서는 유용하지만, 데이터가 많아질수록 count 쿼리 때문에 성능 부담이 생길 수 있다.

---

## 3. Slice

`Slice`는 전체 데이터 개수나 전체 페이지 수는 알지 못하고, **다음 페이지가 있는지만 알 수 있다.**

```java
Slice<Post> posts = postRepository.findAll(pageable);
```

`Slice`는 전체 개수를 세지 않기 때문에 **count 쿼리가 실행되지 않는다.**

대신 요청한 개수보다 1개를 더 조회해서 다음 페이지가 있는지 판단한다.

예를 들어 10개를 요청하면 11개를 조회하고, 11번째 데이터가 있으면 다음 페이지가 있다고 판단한다.

### Slice의 특징

`Slice`는 전체 페이지 수가 필요하지 않은 경우에 적합하다.

대표적으로 **무한 스크롤**이나 **더보기 방식**에서 자주 사용된다.

사용자는 전체 페이지가 몇 개인지 알 필요 없이, 다음 데이터가 있는지만 알면 되기 때문이다.

---

## 4. Page와 Slice 비교

| 구분        | Page       | Slice       |
| --------- | ---------- | ----------- |
| 전체 데이터 개수 | 알 수 있음     | 알 수 없음      |
| 전체 페이지 수  | 알 수 있음     | 알 수 없음      |
| 다음 페이지 여부 | 알 수 있음     | 알 수 있음      |
| count 쿼리  | 실행됨        | 실행되지 않음     |
| 사용 예시     | 일반 게시판 페이지 | 무한 스크롤, 더보기 |

---

## 5. 정리

`Page`는 **전체 데이터 개수와 전체 페이지 수가 필요한 경우** 사용한다.

`Slice`는 **다음 페이지가 있는지만 알면 되는 경우** 사용한다.

따라서 일반적인 게시판처럼 페이지 번호가 필요한 화면에서는 `Page`를 사용하는 것이 좋고, 무한 스크롤이나 더보기 방식에서는 `Slice`를 사용하는 것이 좋다.

---

## 핵심 요약

### Page

* 전체 데이터 개수와 전체 페이지 수를 알 수 있다.
* count 쿼리가 실행된다.
* 일반 게시판 페이지에 적합하다.

### Slice

* 다음 페이지가 있는지만 알 수 있다.
* count 쿼리가 실행되지 않는다.
* 무한 스크롤이나 더보기 방식에 적합하다.

---

# Java Stream API

## 1. 정의

`Stream API`는 Java에서 컬렉션 데이터를 **함수형 방식으로 처리할 수 있게 해주는 기능**이다.

`List` 같은 데이터를 반복문으로 하나씩 처리하는 대신, `filter()`, `map()`, `collect()` 같은 메서드를 사용해서 더 간결하게 처리할 수 있다.

즉, 데이터를 다음과 같은 흐름으로 처리할 수 있다.

```text
전체 데이터
→ 조건에 맞는 데이터만 거르기
→ 필요한 형태로 변환하기
→ 결과로 모으기
```

---

## 2. 자주 사용하는 메서드

| 메서드          | 의미                        |
| ------------ | ------------------------- |
| `stream()`   | 리스트를 하나씩 처리할 수 있는 흐름으로 바꿈 |
| `filter()`   | 조건에 맞는 데이터만 남김            |
| `map()`      | 데이터를 다른 형태로 변환            |
| `forEach()`  | 각각의 요소를 반복 처리             |
| `toList()`   | 결과를 `List`로 변환            |
| `collect()`  | 결과를 원하는 형태로 모음            |
| `sorted()`   | 정렬                        |
| `distinct()` | 중복 제거                     |
| `limit()`    | 개수 제한                     |

---

## 3. 예시 코드

```java
List<String> names = members.stream()
        .filter(member -> member.getAge() >= 20)
        .map(Member::getName)
        .toList();
```

이 코드는 다음과 같은 흐름으로 동작한다.

1. `members` 리스트를 Stream으로 만든다.
2. 나이가 20 이상인 회원만 걸러낸다.
3. 회원 객체에서 이름만 꺼낸다.
4. 결과를 `List`로 만든다.

---

## 4. 쉽게 이해하기

회원 목록이 다음과 같이 있다고 가정해보자.

```text
- 김철수, 18살
- 이영희, 22살
- 박민수, 25살
```

여기서 **20살 이상인 회원만 골라서 이름만 가져오고 싶다**고 해보자.

그러면 처리 과정은 다음과 같다.

1. 회원 목록을 처음부터 하나씩 본다.
2. 나이가 20살 이상인지 확인한다.
3. 20살 이상이면 이름만 꺼낸다.
4. 꺼낸 이름들을 새 목록에 담는다.

이걸 일반 반복문으로 작성할 수도 있지만, `Stream API`를 사용하면 데이터 처리 흐름을 거의 말하듯이 작성할 수 있다.

```java
members.stream()
        .filter(member -> member.getAge() >= 20)
        .map(member -> member.getName())
        .toList();
```

---

## 5. 코드 흐름 분석

### `members.stream()`

```java
members.stream()
```

회원 목록을 하나씩 처리할 수 있는 **Stream 흐름**으로 만든다.

---

### `.filter(member -> member.getAge() >= 20)`

```java
.filter(member -> member.getAge() >= 20)
```

회원 중에서 나이가 20살 이상인 회원만 남긴다.

즉, 조건에 맞지 않는 데이터는 제외된다.

---

### `.map(member -> member.getName())`

```java
.map(member -> member.getName())
```

회원 객체 전체가 아니라, 회원의 이름만 꺼낸다.

즉, `Member` 객체를 `String` 형태의 이름 데이터로 변환한다.

---

### `.toList()`

```java
.toList()
```

처리한 결과를 다시 `List` 형태로 모은다.

---

## 6. Stream API의 핵심 흐름

`Stream API`의 핵심은 데이터를 다음 순서로 처리하는 것이다.

```text
전체 데이터
→ 조건에 맞는 것만 거르기
→ 필요한 형태로 바꾸기
→ 결과로 모으기
```

예를 들어 회원 목록에서 20살 이상인 회원의 이름만 가져오는 경우는 다음과 같다.

```text
회원 목록
→ 20살 이상인 회원만 필터링
→ 이름만 추출
→ 이름 목록으로 저장
```

---

## 7. 주의할 점

### 1. 원본 데이터를 직접 바꾸지 않는다

`Stream`은 원본 데이터를 직접 수정하는 것이 아니라, 데이터를 처리한 결과를 새로 만들어낸다.

즉, 기존 `members` 리스트 자체가 바뀌는 것은 아니다.

---

### 2. 중간 연산과 최종 연산이 있다

`Stream` 연산은 크게 **중간 연산**과 **최종 연산**으로 나뉜다.

| 구분    | 예시                                   | 설명                      |
| ----- | ------------------------------------ | ----------------------- |
| 중간 연산 | `filter()`, `map()`, `sorted()`      | 데이터를 처리하는 과정            |
| 최종 연산 | `toList()`, `collect()`, `forEach()` | Stream을 실제로 실행하고 결과를 만듦 |

`filter()`나 `map()` 같은 중간 연산만 작성하면 실제로 실행되지 않는다.

반드시 `toList()`, `collect()`, `forEach()` 같은 최종 연산이 있어야 실행된다.

---

## 8. 정리

`Stream API`는 컬렉션 데이터를 조건에 따라 거르거나, 다른 형태로 변환하거나, 결과를 모을 때 사용하는 기능이다.

반복문보다 코드가 간결해지고, 데이터 처리 흐름을 읽기 쉬워진다.

---

## 핵심 요약

| 메서드         | 쉽게 정리                     |
| ----------- | ------------------------- |
| `stream()`  | 리스트를 하나씩 처리할 수 있는 흐름으로 바꿈 |
| `filter()`  | 조건에 맞는 것만 남김              |
| `map()`     | 데이터를 다른 모양으로 바꿈           |
| `toList()`  | 처리한 결과를 다시 리스트로 모음        |
| `forEach()` | 하나씩 꺼내서 어떤 작업을 함          |

---

## 한 줄 요약

`Stream API`는 리스트 같은 데이터를 **거르고, 바꾸고, 모으는 작업을 깔끔하게 작성할 수 있게 해주는 기능**이다.

# 객체 그래프 탐색

## 1. 정의

객체 그래프 탐색은 객체가 가지고 있는 연관 객체를 따라가며 데이터를 조회하는 것을 말한다.

JPA에서는 Entity끼리 연관관계가 맺어져 있으면 객체를 통해 다른 객체에 접근할 수 있다.

---

## 2. 예시

```java
Member member = review.getMember();
String name = member.getName();
```

`Review` 객체에서 `Member` 객체로 이동하고, 다시 `Member`의 이름을 가져오는 방식이다.

이처럼 객체의 연관관계를 따라 이동하는 것을 객체 그래프 탐색이라고 한다.

---

## 3. 왜 사용하는가

JPA는 DB 테이블을 직접 다루는 것이 아니라 Entity 객체를 중심으로 데이터를 다룬다.

그래서 SQL처럼 외래 키를 직접 조회하기보다, 객체의 필드를 통해 연관된 객체에 접근한다.

```java
review.getMember().getName();
```

이 코드는 리뷰가 가진 회원 객체로 이동한 뒤, 그 회원의 이름을 가져오는 코드이다.

---

## 4. 장점

* 객체 중심으로 코드를 작성할 수 있다.
* SQL의 join을 직접 생각하지 않아도 연관 객체에 접근할 수 있다.
* 비즈니스 로직을 더 자연스럽게 표현할 수 있다.

---

## 5. 주의할 점

객체 그래프 탐색은 편리하지만, 지연 로딩과 함께 사용할 때 예상하지 못한 쿼리가 추가로 발생할 수 있다.

예를 들어 리뷰 목록을 조회한 뒤 각각의 리뷰에서 회원 정보를 접근하면, 회원을 조회하는 쿼리가 반복해서 발생할 수 있다.

이것이 N+1 문제로 이어질 수 있다.

```java
for (Review review : reviews) {
    System.out.println(review.getMember().getName());
}
```

따라서 연관 데이터를 함께 사용해야 한다면 `fetch join`, `EntityGraph`, `DTO 조회` 등을 고려해야 한다.

---

## 6. 정리

객체 그래프 탐색은 Entity의 연관관계를 따라 객체에서 객체로 이동하며 데이터를 조회하는 방식이다.

JPA를 객체 중심으로 사용할 수 있게 해주는 중요한 개념이지만, 지연 로딩과 N+1 문제를 함께 주의해야 한다.

# @Valid vs @Validated

## 1. 정의

`@Valid`와 `@Validated`는 요청 데이터가 올바른지 검증할 때 사용하는 어노테이션이다.

주로 Request Body로 들어오는 DTO 값이 비어 있지 않은지, 길이가 맞는지, 숫자 범위가 올바른지 등을 검사할 때 사용한다.

---

## 2. 사용 예시

```java
@PostMapping("/members")
public ResponseEntity<?> createMember(@RequestBody @Valid MemberRequest request) {
    return ResponseEntity.ok().build();
}
```

DTO에는 다음과 같이 검증 조건을 작성한다.

```java
public record MemberRequest(
        @NotBlank String name,
        @Email String email,
        @Min(1) int age
) {
}
```

요청 값이 조건을 만족하지 않으면 Controller 로직이 실행되기 전에 검증 오류가 발생한다.

---

## 3. @Valid

`@Valid`는 Java 표준 검증 어노테이션이다.

Spring뿐만 아니라 Bean Validation 표준에서 제공하는 기능이다.

일반적인 DTO 검증에서는 대부분 `@Valid`를 사용하면 된다.

```java
@RequestBody @Valid CreateReviewRequest request
```

---

## 4. @Validated

`@Validated`는 Spring에서 제공하는 검증 어노테이션이다.

`@Valid`보다 확장된 기능을 제공하며, 대표적으로 validation group을 사용할 수 있다.

즉, 같은 DTO라도 상황에 따라 다른 검증 규칙을 적용하고 싶을 때 사용할 수 있다.

```java
@Validated(CreateGroup.class)
```

또한 Controller 클래스나 Service 클래스에 붙여서 메서드 파라미터 검증을 활성화할 때도 사용된다.

---

## 5. @Valid와 @Validated 비교

| 구분               | @Valid           | @Validated      |
| ---------------- | ---------------- | --------------- |
| 제공 위치            | Java 표준          | Spring 제공       |
| 기본 DTO 검증        | 가능               | 가능              |
| Validation Group | 직접 지원하지 않음       | 지원              |
| 주 사용 위치          | Request Body DTO | 클래스, 메서드, 그룹 검증 |
| 일반적인 사용          | 많이 사용            | 특수한 상황에서 사용     |

---

## 6. 언제 무엇을 쓰는가

단순히 Request Body의 값을 검증할 때는 `@Valid`를 사용하면 충분하다.

```java
@RequestBody @Valid CreateMissionRequest request
```

검증 그룹을 나누거나, Spring의 메서드 파라미터 검증 기능을 사용해야 한다면 `@Validated`를 사용한다.

```java
@Validated
@RestController
public class MemberController {
}
```

---

## 7. 정리

`@Valid`는 일반적인 요청 DTO 검증에 사용하고, `@Validated`는 Spring에서 제공하는 확장 검증 기능이 필요할 때 사용한다.

대부분의 API 요청 검증에서는 `@Valid`를 사용하면 되고, 상황별 검증 그룹이 필요할 때 `@Validated`를 사용하면 된다.
